## 01. 데이터베이스란?

### 데이터-베이스

> **기록하고 조합하는 모든 데이터들의 총합**

### 일상에서 활용되는 데이터베이스

- 통장의 입·출금 기록
- 도서관에서의 도서 검색

### 은행 결제내역 조회 — 4단계

1. **1단계**: 은행앱 열기
2. **2단계**: 은행 서버에 내 계좌정보로 접속
3. **3단계**: 접속정보 확인 / 내 계좌정보 데이터 베이스 검색
4. **4단계**: 원하는 결제 내역 확인

### 도서관 관리 프로그램 — 도서관 관리를 위해 필요한 정보

- 도서관이 해당 책 보유 여부
- 책의 정보 (책 제목, 작가, 출판사)
- 현재 대여 중인지?
- 누가 대여했는지?
- 대여중이라면, 언제 빌려갈 수 있는지

## 02. 데이터베이스 기능

### 데이터 베이스의 기본 기능 (CRUD)

```
[저장] → [검색] → [수정] → [삭제]
```

### 저장 (Insert)

- 데이터의 저장
- 상품 정보
- 사진, 가격, 상세설명

### 데이터베이스 저장 (예시)

가격, 상표명, 사진, 상세 설명 등의 **상품 판매에 필요한 정보를 저장**

### 데이터베이스 테이블 (상품 예시)

| 상품번호 | 상품종류 | 가격 |
|---|---|---|
| 1 | 상의 | 50000 |
| 2 | 청바지 | 80000 |
| 3 | 신발 | 70000 |
| 4 | 벨트 | 10000 |

### 데이터베이스 테이블 (회원정보 예시)

| 상품번호 | 이름 | 아이디 | 주소 |
|---|---|---|---|
| 1 | 홍길동 | A1 | 서울 |
| 2 | 철수 | B2 | 부산 |
| 3 | 영희 | C3 | 광주 |
| 4 | 광수 | D4 | 대전 |

### 데이터베이스 테이블의 본질

> **데이터를 분류하기 좋게 저장되어 있는 형태**
> - 사람이 분류하기 쉽게 저장하는 것이 중요
> - 서비스를 어떻게 운영할 것인가의 핵심 요소

### 케이스 예시: "신발이 매진 되었을 때"

| 상품번호 | 상품종류 | 가격 |
|---|---|---|
| 1 | 상의 | 50000 |
| 2 | 청바지 | 80000 |
| **3** | **신발** | **70000** |
| 4 | 벨트 | 10000 |

상품의 라이프사이클을 데이터에 반영해야 함:

```
[상품조회] → [결제] → [환불]
- 신발 상품정보   - 신발 구매 진행   - 판매 수량 X
- 가격           - 판매 수량 X     - 구분할 로직 없음
```

→ 단순 가격/종류 컬럼만으로는 부족. 매진 여부를 표현하는 컬럼이 필요.

### 검색 (select, find)

```
[저장]  [검색▼]  [수정]  [삭제]
```

데이터베이스의 기본 기능 중 하나.

### 검색의 활용 — 특정 조건

특정 조건을 활용해 저장되어 있는 데이터들을 데이터베이스로 부터 분류해 놓은 키워드:

- 신상품 입고
- 핫딜
- 가격 낮은순
- 추천상품

### 신상품 입고 목록 — 흐름

```
[신상품 테이블] → [조회] → [결과]
- 신상품만 저장    - 신상품 테이블    - 조회 결과 반환
                  - 전체 목록 조회
```

### Query, Query 문

```
[클라이언트] → [DB]
        Query
```

> **Query = 데이터베이스 명령어**

### 신상품 입고 목록 갱신 — 흐름

```
[신상품 테이블] → [삭제 이동] → [신규 상품 등록] → [결과]
- 기존에 있던     - 신상품이 더 이상   - 새로운 신상품을    - 새로운 신상품만
  신상품 목록     아니므로 삭제하거나   다시 DB 등록        결과 조회
  데이터          다른 테이블로 이동
```

### 데이터 베이스 정렬

> **특정한 값을 기준으로 데이터를 정렬할 수 있다**

### 신상품 입고 목록 갱신 (정렬 흐름)

```
[신상품 등록] → [조회] → [정렬] → [결과]
- 신상품 등록    - 신상품 테이블    - 최신 등록 순으로    - 정렬 조건에 맞게
- 등록 일자       목록 조회 요청     20개만 조회           앱/웹 에 게시
  함께 기록                         - 8 ~ 10월 상품만
                                    조회
```

### DB가 우리에게 오기 까지

```
[웹 브라우저/스마트폰] → [서버 조회 요청] → [DB query 조회]
        ↑                                          ↓
  [조회 결과를 전달] ←──────────────────────────────
```

> **메모 — DB**: 데이터베이스 Database 줄임말

### 데이터 베이스 설계

> **원하는 목적에 맞게 초기에 구성을 잘 하는 것이 중요**

### 수정 (update)

```
[저장]  [검색]  [수정▼]  [삭제]
```

### 컴퓨터의 데이터 타입

| 분류 | 타입 | 설명 |
|---|---|---|
| 숫자 | Int, float, double | 정수, 소수 형태 숫자 |
| 문자 | String | 문자열 (이름, 주소) 형태의 데이터 |
| 문자 | Boolean | 참/거짓 의 데이터 형태 |
| 시간 | Time | Datetime 날자, 시간 형태 |

### 신발이 매진 되었을때 — 매진 컬럼 추가

| 상품번호 | 상품종류 | 가격 | 매진 |
|---|---|---|---|
| 1 | 상의 | 50000 | False |
| 2 | 청바지 | 80000 | False |
| **3** | **신발** | **70000** | **True** |
| 4 | 벨트 | 10000 | False |

→ Boolean 타입 컬럼으로 매진 여부 관리.

### 삭제 (delete)

```
[저장]  [검색]  [수정]  [삭제▼]
```

### 데이터베이스 삭제 (물리 삭제)

| 상품번호 | 상품종류 | 가격 | 매진 |
|---|---|---|---|
| 1 | 상의 | 50000 | False |
| 2 | 청바지 | 80000 | False |
| ~~3~~ | ~~신발~~ | ~~70000~~ | ~~True~~ |
| 4 | 벨트 | 10000 | False |

### 신발이 매진 되었을 때 — 논리삭제 (vs 물리삭제)

| 상품번호 | 상품종류 | 가격 | 매진 |
|---|---|---|---|
| 1 | 상의 | 50000 | False |
| 2 | 청바지 | 80000 | False |
| **3** | **신발** | **70000** | **True** |
| 4 | 벨트 | 10000 | False |

→ 데이터를 실제로 지우지 않고 매진=True 플래그로 처리 (논리 삭제).

## 03. 데이터베이스와 서버 — "서버가 터졌어요"

### 서버오류

```
[클라이언트] —🚫→ [DB]
```

> **프론트엔드 ↔ 백엔드 — 클라이언트와 서버가 통신이 불가능해진 상황**

### 다양한 케이스의 서버 오류

- 트래픽 과부하
- 백엔드 코드의 버그
- 데이터베이스 접속 오류

### 서버오류 — 트래픽 과부하

```
[클라이언트] ┐
[클라이언트] ├──→ [DB]  (다수 동시 접속)
[클라이언트] ┤
[클라이언트] ┘
```

> **메모 — 트래픽**: 서버에 접속할때 일어나는 통신양을 의미

### 서버오류 — DB 오류

```
[클라이언트] → [웹 서버] —응답 불가→ [DB]
```

**서버에 접속은 가능하지만 DB 와의 연결이 안되는 경우**

## 05. API와 데이터베이스

### API

> **프로그램끼리(Code 끼리) 대화를 하도록 도와주는 일**

### API의 풀네임

**Application Programing Interface**

### API 요청 (로그인 예시)

```
[ID/PW 로그인] → [로그인 API]
       ↑                ↓
[로그인 성공]  ← [로그인 정보 DB 조회]
```

### 로그인 API — endpoint와 요청

```
[클라이언트] —— Shoppingmall.com/login ——→ [DB]
              로그인 요청
              아이디/비밀번호 전송 (POST)
```

### GET API — 국가별 사이트 정보

같은 endpoint 패턴이지만 국가코드에 따라 다른 데이터를 가져옴:

- `https://www.apple.com/kr/` — 한국어 사이트 정보를 가져옴 (iPad 이미지)
- `https://www.apple.com/jp/` — 일본어 사이트 정보를 가져옴 (iPhone 14 이미지)

```
https://www.apple.com/국가코드/
```

### API 정의 (재정의)

> **네트워크를 통한 기능호출**

### Open API — 사례: 주소 찾기

행정안전부에서 제공하는 Open API 를 활용하여 만들어진 것 (예: 주소 찾기 기능).

오픈API 상세 정보 예시:
- 분류체계: 통신 - 우정
- 관리부서명: 주소정책과
- API 유형: LINK
- 활용신청: 7544
- 등록: 2016-10-31
- 데이터포맷: JSON+XML
- URL: http://www.juso.go.kr/openIndexPage.do
- 비용부과여부: 무료

## 06. 데이터베이스의 종류와 선택

### DBMS

> **Database Management System — 데이터베이스 운영관리를 위한 소프트웨어**

### DBMS 의 종류

| DBMS | 제작사 |
|---|---|
| MySQL | Oracle |
| MariaDB | MariaDB |
| PostgreSQL | PostgreSQL |
| Oracle | Oracle |

### 데이터 베이스의 종류 — 관계형 vs Nosql

- **관계형 (RDB)**: Primary key 라는 고유값을 활용해 연결에 유리한 관계형
  - 예: B신발을 구매한 A고객. (B신발 데이터 + A 고객 데이터)
- **Nosql**: 기록이 단순하고 대용량 처리에 유리
