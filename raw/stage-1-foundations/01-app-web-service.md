## 01. 우리가 사용하는 웹/앱의 시작

### 웹/앱 서비스의 개요

```
명함촬영 → 명함 사진 정보전송 → DB 직접입력 → 디지털 명함
```

**디지털 명함 앱 서비스를 기준으로 웹/앱 서비스를 만들기 위한 내용들을 파악해 보자**
(전체 강의의 러닝 예제로 "디지털 명함 앱"이 사용됨)

## 02. 웹 주소의 시작, http의 정체

### 통신이란?

> **컴퓨터 간의 Communication (대화)**

1. 컴퓨터와 컴퓨터 간의 대화
2. 스마트폰과 서버의 대화
3. 나의 스마트폰과 친구 스마트폰의 메신저

### 통신의 예시

- 배달앱 ⇄ 치킨집: "후라이드 치킨 주문" → "주문 OK! 20분 기다려줘~"
- 명함앱 ⇄ 서버: "명함정보 전달! 저장요청" → "저장되었어! 디지털명함"

### 통신의 원칙

> **기기간의 요청과 응답으로 이루어진 대화. 1요청, 1응답의 원칙을 가진다**

### HTTP — Hyper Text Transfer Protocol

- **H** hyper — 기기(스마트폰)에 상관없이 문자로 통신할 수 있는 방식
- **T** text — 운영체제와 관계없이 문자로 통신할 수 있는 방식
- **T** transfer — 웹 브라우저에 상관없이 통신할 수 있는 방식
- **P** protocol

> **메모 — Protocol**: 통신 방식에 있어서 특정한 룰을 가지고 이대로 통신을 하자는 약속된 형태의 방식을 의미한다.

### Hyper text 의 종류

**JSON 의 형태 예시 — 명함**

```json
{
  "id": 1,
  "name": "홍길동",
  "company": "(주)엘리스",
  "phone": "010-1234-5678"
}
```

**XML 의 형태 예시 — 음반 발매 정보**

```xml
<CD>
  <TITLE>Still got the blues</TITLE>
  <ARTIST>Gary Moore</ARTIST>
  <COUNTRY>UK</COUNTRY>
  <COMPANY>Virgin records</COMPANY>
  <PRICE>10.20</PRICE>
  <YEAR>1990</YEAR>
</CD>
```

> **메모 — Parsing(파싱)**: 문자를 특정한 형태로 잘라서 필요에 따라 사용하는 것을 의미한다. 옆의 JSON 명함 형태를 파싱한다면 명함의 이름, 회사, 번호를 분류하여 필요에 따라 사용할 수 있게 자른다는 의미이다.

## 03. WAS (Web Application Server) 의 구조

### 일반적인 Web Application Server 의 기본 구조

```
[클라이언트]  →요청 Request→  [Web Server]  ⇄  [Back-end]  ⇄  [DataBase]
(웹 브라우저,    ←응답 Response←  (http 통신)        동작코드      (각종 정보 저장)
 스마트폰)
```

### 통신의 흐름 (명함 저장 예시)

```
[명함촬영] ⇄ [Web Server (http 통신)] ⇄ [Hypertext 파싱 후 저장할 공간을 찾아 전달] ⇄ [정보를 저장]
        명함정보 저장 요청 →
        ← 명함정보 저장 여부
                                              정보 저장 결과 전달 →
```

> **메모 — Endpoint**: 웹 서버의 주소를 가르키는 문자를 의미한다. 우리가 브라우저에서 입력하는 www.google.com, www.naver.com 등의 주소를 가르키는 url 을 Endpoint 라고 한다

## 04. Front-end (클라이언트)

### Front-end 정의

- 서비스에서 처음 만나는 화면으로 서비스의 의도대로 사용자를 유도
- 서버의 기능과 관계없이 단독적인 기능들의 모음 (ex. 카메라 촬영)

### Front-end 의 주요 특징

1. UI/UX (화면과 사용자 경험 구성)
2. 독립적인 기능들
3. 서버에 요청하기 위한 통신 기능
4. 사용자 패턴 분석을 위한 Analytics 구현

### UI (User Interface)

- 사용자가 처음 만나는 화면
- 디자인적 요소들을 담고 있는 화면
- 사용자의 서비스 사용 편의성을 위한다

> **메모 — GUI**: Graphics User Interface 의 줄임말으로 단어 그대로 그래픽을 활용한 사용자 환경을 의미한다. 현대에는 그래픽 환경이 압도적으로 많기 때문에, GUI, UI 를 구분 짓지 않고 UI 라는 단어로 통용된다.

### UX (User Experience)

> **서비스의 성격에 따라 사용자에게 편리한 경험을 제공하는 것**

예: 카메라 촬영 기능에 도달하는 두 가지 경로
- 로그인 → 메인화면 → 메뉴 탐색 → 기능 → 카메라 촬영 (긴 경로)
- 로그인 → 메인화면의 하단 버튼 → 카메라 촬영 (짧은 경로)

### 반응형 화면

데스크탑, 태블릿, 스마트폰 등 다양한 화면에 대응.

**반응형 화면의 특징**
1. 다양한 사이즈의 스크린을 커버
2. 동일한 정보 제공 환경을 제공

### UI — Grid System

**Grid System = 모바일 : Component System**

- 반응형을 위해 일관된 규칙성을 적용
- 화면비에 대한 계산이 필요
- 작은 화면에서의 정보제공 손실 고려

(예시: 2 Columns / 3 Columns / 4 Columns / 6 Columns 단위로 화면 분할)

> **메모 — Grid System**: 페이지 컨텐츠를 수학적 논리를 가지고 일관되게 구조화 시켜서 디자인하도록 하는 시스템을 말한다

### 프론트앤드 에서의 통신 (http 통신의 함수들)

| 함수명 | 기능 |
|---|---|
| **GET** | 기존의 저장된 정보, 페이지를 읽어온다 |
| **POST** | 정보의 저장 및 전달을 위해 서버에 전달하는 역할 |
| **PUT, PATCH** | 기존의 저장된 정보의 내용을 수정하는 역할 |
| **DELETE** | 저장된 정보를 삭제 |

> **메모 — PUT, PATCH**: PUT 은 정보를 수정하는 http 통신의 함수이지만 보안적인 이슈가 꾸준히 재기되고 있어 사용할 때 많은 주의를 필요로 한다. 이를 보완해서 나온 것이 PATCH 이다.

### 프론트앤드에서의 POST

```
[명함촬영] ⇄ [Server]
   저장할 명함정보 전달 (POST) — 이름, 회사명, 전화번호 →
   ← 명함정보 저장 여부 결과
```

### 프론트앤드에서의 GET

```
[로그인 후] ⇄ [Web Server]
   저장할 명함정보 요청 (GET) →
   ← 저장된 명함정보 전달
```

### 일상에서의 GET

- 로그인 후 나의 이름이 적힌 Welcome 페이지 문구 ("헬로빗님 다시 만나서 반가워요")
- 쇼핑몰에서의 제품 상세 페이지 (가격, 이미지 등 표시)

### PUT, PATCH

```
[명함정보] ⇄ [Web Server]
   명함정보 수정 (PUT, PATCH) →
   ← 수정된 명함정보 전달
```

예시: 명함정보 수정 요청, 비밀번호 변경 요청

### DELETE

```
[기존명함정보] ⇄ [Web Server]
   명함정보 삭제 (DELETE) →
   ← 삭제결과 전달
```

예시: 명함정보 삭제, 서비스 계정 삭제 (계정 탈퇴 — "탈퇴 후 복구할 수 없습니다. 탈퇴하기")

## 05. 동기 (Synchronous) 와 비동기 (Asynchronous)

### 동기와 비동기

- **동기**: 통신의 결과와 함께 움직임
- **비동기**: 통신의 결과와 상관없이 움직임

### 동기(Synchronous) 사례

```
[로그인 후] ⇄ [Web Server]
   저장된 명함정보 요청 (GET) →
   ← 저장된 명함정보 전달
```

- 강의 목록에서 상세페이지로의 이동
- 저장된 명함 정보를 가져오는 페이지
- 일반적인 웹 페이지의 이동
- 정보와 함께 제공되어져야 하는 화면

(예: "강의 컨텐츠 목록 페이지" → "강의 상세 소개 페이지")

### 비동기(Asynchronous) 사례

> **통신의 결과와 상관없이 독립적인 결과를 가져옴**

예) 페이지의 이동이 없는 댓글 달기, SNS 피드백

```
[영업사원] → [명함전달] → [수령후 입력] → [디지털 명함]
    ↓
[영업활동]   (디지털 명함이 없어도 영업 활동은 독립적으로 가능)
```

## 06. 백엔드와 API

### 백엔드

> **서버 측의 동작 Code 비즈니스 로직(Business logic)을 처리**

> **메모 — 비즈니스 로직**: 비즈니스 로직이란 컴퓨터 프로그램에서의 데이터를 생성, 표시, 저장, 변경하는 모든 내용들을 지칭한다

### 백엔드 와 비즈니스 로직 예시

```
[Web Server (http 통신)] ⇄ [Hypertext 파싱 후 저장할 공간을 찾아 전달] ⇄ [정보를 저장]
                                                                      ← 정보 저장 결과 전달
```

활용 예시:
- 명함 정보의 저장 및 변경
- 쇼핑몰에서의 상품 구매 및 결제
- SNS 회원 가입

### 백엔드 에서의 통신 (http 통신의 함수들)

| 함수명 | 기능 |
|---|---|
| **GET** | 기존의 저장된 정보, 페이지를 읽어온다 |
| **POST** | 정보의 저장 및 전달을 위해 서버에 전달하는 역할 |
| **PUT, PATCH** | 기존의 저장된 정보의 내용을 수정하는 역할 |
| **DELETE** | 저장된 정보를 삭제 |

> **메모 — PUT, PATCH**: PUT 은 정보를 수정하는 http 통신의 함수이지만 보안적인 이슈가 꾸준히 재기되고 있어 사용할 때 많은 주의를 필요로 한다. 이를 보완해서 나온 것이 PATCH 이다.

### 프론트앤드와 백앤드 통신

```
프론트엔드 요청     →     백엔드 응답
GET      ↓              GET
POST     ↓              POST
PUT,PATCH ↓             PUT,PATCH
DELETE   ↓              DELETE
```

> **메모**: 프론트가 GET 으로 요청을 하면 백엔드도 반드시 GET 으로 호응해야 한다

### REST API

```
약속된 주소: https://school.elice.io/courses/26028/info
                       ↓
[프론트엔드(요청)] — GET — [백엔드(응답)] — GET
```

**약속된 지정된 주소를(URI) 가지고 있고, 각 주소에 기능 (함수) 를 가지고 있는 형태**

> **메모 — URI vs URL**:
> - URI 는 서비에서의 특정 리소스 식별자를 지칭하고
> - URL 은 우리가 흔히 사용하는 웹 사이트의 주소를 뜻한다
> - URI 가 URL 보다 더 큰 의미라는 것만 기억하고 넘어가자
