# Operating System (운영체제)

## 한 줄 정의

> **"하드웨어를 포함한 리소스를 제어하고, 프로그램을 실행해주는 시스템 소프트웨어"** (강의 본문)

[[컴퓨터-기초]]에서 짧게 언급됐던 OS의 알맹이 전체. 이 페이지가 OS 시리즈(8강 중 OS 이론 4강) 진행 중 허브 역할.

## 핵심 아이디어

### 왜 OS가 필요한가 — 사장 비유 ⭐

```
[작은 회사]  사장 1명 ──→ 직원 몇 명 다 이해·통제 가능
[큰 회사]    사장 1명 ──→ 직원 너무 많음 → 다 통제 불가
                          ↓
                    "관리자" 계층 필요
```

대응:
- 옛날: 하드웨어 종류 적고 전문가만 사용 → 굳이 OS 필요 없음
- 현재: 하드웨어 종류 많고 다양한 용도 → **OS가 중간 관리자 역할** 필수

### 리소스 (System Resource)

- 시스템 자원 — 어떤 목적에 이용할 수 있는 **하드웨어 포함 모든 것**
- 일상 예시: Windows "리소스가 부족하여 이 작업을 완료할 수 없습니다. 데이터를 더 적게 선택하거나 다른 응용 프로그램을 닫으십시오"
  - 여기서 "리소스" = 메모리
- **핵심 규칙**:
  - 각 응용프로그램마다 **할당된 리소스가 있음**
  - 응용프로그램은 다른 응용프로그램의 리소스를 **강제로 사용할 수 없음**

→ 즉 OS는 리소스를 **할당·격리**하는 중재자.

## OS의 4대 기능

```
프로세스 관리 → 메모리 관리 → 파일 시스템 관리 → 네트워크 관리
```

→ 시리즈 다음 4강(프로세스·메모리·파일 시스템·…) 매핑 그대로. 이 페이지는 도입부 정의만 담고, 본격 알맹이는 분기 페이지에서 누적 예정.

### 1. 프로세스 관리

> **프로세스 = 실행 중인 프로그램** (Program in execution)

- **리소스 할당**: 프로세스에 필요한 리소스 분배
- **상태 관리**: 각 프로세스의 상태 추적
- **멀티 프로세싱**: 동시에 여러 프로세스 실행

→ 다음 강 ([수업자료] 프로세스) 진입 시 별도 페이지로 분리 예정.

### 2. 메모리 관리

> CPU에서 실행될 수 있도록 메모리에 정보를 올리거나 내림.

- 메모리에 프로그램을 올림 / 프로세스에 필요한 메모리 할당
- **가상 메모리**: 실제 메모리가 적더라도 프로세스를 실행할 수 있도록 도와주는 기술

#### 가상 메모리 = 안심 번호 비유 ⭐⭐

```
[택배기사] ─→ [통신사: 안심번호] ─→ [나: 실제번호]
            (가상 주소)              (실제 주소)
```

- 실제 주소가 아닌 **가상의 주소**로 메모리 관리
- 장점 2가지:
  - **정보 보호**: 프로세스가 실제 주소를 모르기에 메모리 주소를 악용한 **침범 불가능** (= 다른 프로세스 침범 차단)
  - **연속 주소 제공**: 프로세스가 보는 주소는 연속된 주소 → **효율적 처리** 가능 (실제 메모리는 조각나도 OK)

→ 다음 강 ([수업자료] 메모리) 진입 시 별도 페이지로 분리 예정.

### 3. 파일 시스템 관리

> "어떤 방식을 사용하던지 프로그램에서 고민하지 않고 읽고 쓰게 해주는 기능" (강의 본문)

- **파일 시스템** = 저장장치에 파일을 읽고 쓰는 방법
- OS가 추상화해서 프로그램은 디스크 종류·구조 신경 X

→ 다음 강 ([수업자료] 파일 시스템 + [강의자료] 리눅스 파일 시스템) 진입 시 별도 페이지로 분리 예정.

### 4. 네트워크 관리

- **프로토콜** = 네트워크 상의 컴퓨터끼리 통신하는 규칙
  - → [[HTTP-프로토콜]]은 그 중 하나 (HTTP). OS는 다양한 프로토콜(TCP/UDP/IP 등) 지원
- **포트 관리**: 이미 사용 중인 포트는 사용할 수 없도록 제어
- 프로그램이 네트워크를 사용할 수 있게 하부 작업 처리

## 컴퓨터 하드웨어 복습 (1강 도입부)

[[컴퓨터-기초]]의 알맹이 보강:

| 구성 | 설명 |
|------|------|
| **CPU** | Central Processing Unit, 중앙 처리 장치. 데이터·명령어를 처리하는 제어 장치. 정보는 메모리에서 가져옴 |
| **메모리** | CPU가 처리할 데이터·명령어 저장. 주소 부여됨(주소 참조로 데이터 찾음). 보조기억장치보다 빠르지만 **전원 차단 시 내용 삭제** (= RAM) |
| **주변장치 - 입력** | 마우스, 키보드 |
| **주변장치 - 출력** | 모니터, 스피커, 프린터 |
| **주변장치 - 보조기억장치** | SSD, HDD (전원 차단돼도 내용 유지) |

→ [[컴퓨터-기초]] 페이지에도 누적.

## OS의 종류

| OS | 제작·특징 |
|----|----------|
| **Windows** | Microsoft, 데스크탑, 일반 사용자에게 적합, **가장 높은 점유율** |
| **macOS** | Apple, Unix 기반, **유닉스 계열 터미널 명령어 사용 가능** → 개발 환경에 적합, 상용 |
| **Linux** | **무료·오픈소스**, 개발·서버에 적합, 기업·개인이 수정·판매 가능, 우분투·데비안 등 다양한 배포판. **안드로이드도 Linux 기반** |

→ [[현업-용어]] 오픈소스 섹션과 직접 연결 (리눅스가 오픈소스 대표 사례).

### 컨테이너화 (강의 마지막 슬라이드, Docker 추정)

> 실행에 필요한 모든 환경과 패키지를 포함하여 하나로 묶는 기술. **내가 사용하는 OS에 상관없이 동일한 개발환경**을 구성할 수 있음.

- **사용자 적용 직결**: 프로젝트윤365 1인 컨설팅에서 **클라이언트 환경 차이로 인한 디버깅 시간** 절감
- 강의에선 한 슬라이드로 짧게 등장 → 후속 학습 필요

## 왜 중요한가

- **모든 상위 계층의 토대** — 앱·웹·클라우드·AI 다 OS 위에서 돌아감
- **사용자 배경상 작동 지식은 있음**: macOS·iOS 일상, Linux 추상화 위에서 Cloudflare·Claude Code 사용 — 이번 페이지는 **명시적 멘탈 모델 정리**
- **AI 트랙 진입 후에도 자주 부딪힘**: GPU 메모리 관리·프로세스 관리·컨테이너(Docker) 등 모두 OS 개념의 응용

## 자주 헷갈리는 것

- **OS = Kernel?**: 엄밀히 다름. 커널은 OS의 핵심 부분이고, OS는 커널 + 셸 + 시스템 도구 등 포함. 강의에선 거의 동의어로 사용 가능 (리눅스 시리즈에서 본격 등장 예정)
- **메모리 ≠ 저장장치**: 메모리(RAM)는 전원 끄면 사라짐, 저장장치(SSD·HDD)는 유지. 둘 다 "데이터 저장"이지만 역할 완전히 다름
- **가상 메모리 ≠ 가짜 메모리**: 진짜 메모리를 추상화한 주소 체계. RAM이 부족할 때 디스크 일부를 메모리처럼 쓰는 페이징(swap)은 그 응용
- **프로세스 vs 프로그램**: 프로그램은 디스크에 있는 파일, 프로세스는 메모리에 올라가 실행 중인 상태

## 프로젝트윤365 / 인스타 콘텐츠 적용

- **"가상 메모리 = 안심 번호" 1컷** ⭐ — 강의 핵심 비유, 비전공자에게 즉시 와닿음
- **"사장 혼자 다 못 봐서 OS가 생긴 거"** — OS 필요성 1컷
- **"Windows·macOS·Linux 한 줄 비교"** — 비전공자 친화 콘텐츠
- **"Docker가 1인 컨설팅 클라이언트 환경 통일에 왜 효율적인가"** — 프로젝트윤365 자산 후보

## 🙋 은미 정수 (2026-05-29) — OS 기초 + 가상메모리 + 추상화

### OS vs 개발 언어 — 헷갈리지 말 것

| 기기 | OS | 개발 언어 |
|---|---|---|
| 맥북 | **macOS** | Swift·Objective-C |
| 일반 PC | **Windows** | C#·C++·Java |
| 아이폰 | **iOS / iPadOS** | **Swift** |
| 갤럭시·픽셀 | **Android** | **Kotlin**·Java |

→ Swift는 OS가 아니라 **iOS 앱 만드는 언어**. Android는 Kotlin.

### 리소스 (Resource) — 작업 관리자

**리소스 = OS가 분배하는 자원** (CPU·메모리·디스크·네트워크).

- 컴퓨터 = 제한된 자원
- 백그라운드에서 카톡 동기화·백신 검사 등이 리소스 갉아먹음
- 작업 관리자: **Mac = `Cmd+Option+Esc`** / Windows = `Ctrl+Shift+Esc`
- 너무 많으면 → 컴퓨터 버벅·강제 종료

→ **OS 핵심 역할 = 리소스를 프로그램에 싸우지 않게 공평 분배**.

### 프로세스 — 시분할 시스템

내 눈에 인스타·유튜브·카톡 **동시 실행 ⚠️ 착각**.

**실제**: CPU 1코어는 한 번에 1개만. **빛의 속도로 "0.001초씩 돌아가며"** 처리.

→ 1초에 수십억 번 전환 → 동시처럼 보임. **시분할(Time-sharing)**.

### 가상메모리 — 책상·서랍장 비유 ⭐

**책상 = 진짜 메모리(RAM) / 서랍장 = 디스크(SSD)**

```
국·영·수·과·사 5권 펼치고 싶음
→ 책상 좁아서 3권만
→ OS: 안 보는 과학·사회 → 서랍장(가상메모리)으로
→ 과학 펴면 → 수학을 서랍장으로 보내고 과학을 책상으로
```

**용량 한계**:
1. 창고(디스크) 크기
2. **CPU 주소 한계** — 32비트 = 4GB / **64비트 = 16EB** (사실상 무제한이지만 결국 디스크 한계)

**핵심**: 위치만 다를 뿐 결국 **내 컴퓨터 자원**. RAM = 빠르고 비쌈 / 디스크 = 느리고 큼.

### 가상메모리 vs 캐시 — 반대 방향 ⭐

| | 가상메모리 | 캐시 |
|---|---|---|
| **방향** | **아래로 확장** (느린 디스크) | **위로 확장** (초고속 메모리) |
| **목적** | 용량 부족 해결 | 속도 향상 |
| **비유** | **옷장 서랍에 임시 저장** | **모니터 옆 포스트잇** |
| **속도** | RAM보다 느림 | RAM보다 빠름 |

→ 가상메모리 = "**살기 위한 몸부림**" / 캐시 = "**짜릿한 쾌속 질주**".

### 가상메모리 ↔ 비동기 상관관계 ⭐⭐

가상메모리에서 데이터 가져오는 순간 = **페이지 폴트 (Page Fault)** + **I/O 작업** (느림).

**동기 방식이면**: CPU가 디스크 응답 기다리며 **멈춤** (Blocking) → 컴퓨터 렉

**비동기 방식**: CPU가 디스크 요청 던져두고 **다른 일 처리** → 데이터 도착 시 알림(인터럽트) → 매끄럽게 진행

→ **가상메모리의 느려짐을 비동기로 감춤**. OS 효율성의 핵심.

### 추상화 (Abstraction) — 정확한 뜻 ⭐

**개발에서 추상화 = "복잡한 알맹이는 숨기고 쓰기 편한 껍데기만"**.

| | 알맹이 (숨김) | 껍데기 (드러냄) |
|---|---|---|
| **자동차** | 엔진·피스톤·연료 분사 | **가속 페달** |
| **파일 시스템** | 자석판·전기 신호·물리 주소 | **노란 폴더·파일 아이콘** |
| **결제 버튼** | 카드사·은행 통신·보안 | **결제하기 버튼** |

→ 인문학 "추상=뜬구름"과 정반대. 개발 "추상=**복잡한 거 숨기고 쓰기 쉽게 정제**".

**한 줄**: 알맹이 뺀 껍데기 = 너 정답.

### 파일 시스템 = 디스크 추상화

- **디스크 민낯**: 0/1 격자판·물리 주소
- **파일 시스템**: 그 위에 OS가 만든 **"노란 폴더·아이콘 파일"** 껍데기
- 우리가 마우스 우클릭으로 폴더 만들 때, 뒤에서 OS가 물리 주소 쪼개서 방 만듦

### 캐시 삭제 권고 이유

1. **낡은 데이터 버그** — 사이트 리뉴얼했는데 옛날 로고 캐시가 자꾸 떠서
2. **비대화** — 캐시가 너무 많아져 도리어 느려짐

→ 삭제 직후 첫 접속은 살짝 느려짐. 단 최신 데이터·정상 동작 확보.

## 🙋 은미 정수 2 (2026-05-29) — 프로세스·스레드·스케줄링 (Day 2 01·02강)

### OS = 생태계 ✅

하드웨어 = 화성 (황무지) / OS = 지구 (생태계). iOS·Android 생태계처럼 OS가 하드웨어 위에 사회 구축.

### 프로세스 vs 스레드 — 아파트 비유

- **프로세스 = 독립 아파트** (101호 카톡 / 102호 유튜브) — 서로 못 봄
- **스레드 = 한 아파트 안의 방들** — 거실·주방 공유 (한 카톡 안의 채팅창·다운로드)

**관계**: **1 프로세스 = 최소 1개 이상의 스레드**. 스레드는 프로세스의 자식.

**시분할 ≠ 메모리 공유**:
- 시분할 = CPU **시간**을 쪼개 쓰는 것
- 프로세스끼리 메모리는 여전히 격리

### 메모리 4영역 ⭐

| 영역 | 역할 | 비유 |
|---|---|---|
| **코드 (Code)** | 명령어 (소스코드 자체) | 책의 본문 |
| **데이터 (Data)** | 프로그램 켜진 동안 살아있는 전역변수 | 등장인물 이름 (안 바뀜) |
| **힙 (Heap)** | 실행 중 동적 할당 (개발자 직접 관리) | 자유 변형 공간 |
| **스택 (Stack)** | 함수 실행 시 잠깐 쓰고 끝나면 칼같이 사라짐 | **단기 기억상실 방** ⭐ |

### 스택 = 단기 기억상실 ⭐

너 정답. 컴퓨터의 **임시 작업대**.

**4가지 비유** (LIFO = Last In First Out):
1. **인스타 스토리 필터** — 글씨 → 고양이 → 카메라 역순 지워짐
2. **햄버거집 주문서** — 마지막 콜라부터 처리
3. **뒤로 가기 버튼** — 가장 마지막에 연 페이지부터 닫힘
4. **뷔페 접시 타워** — 위에서부터 가져가고 쌓음

**왜 4영역에 들어가나?**:
- **자동 청소** — 함수 끝나면 OS가 알아서 깔끔히 지움
- 다른 영역(힙)은 개발자가 직접 지워야 함 → 깜빡하면 **메모리 누수**

### 기술 스택 vs 메모리 스택 — 동음이의어 ⚠️

| | 기술 스택 (Tech Stack) | 메모리 스택 (Memory Stack) |
|---|---|---|
| **정체** | 기술들의 조합 (추상적) | RAM 속 실제 공간 (물리적) |
| **예** | "Node.js+React 스택" | 함수 끝나서 스택 영역 지워짐 |
| **비유** | 요리사 레시피 목록 | 주방의 임시 도마 |

→ 단어만 같음.

### 5상태 전이 — 식당 손님 비유 ⭐

| 상태 | 식당 비유 |
|---|---|
| **생성 (New)** | 식당 앞 도착 + 대기 명단 |
| **준비 (Ready)** | 줄 서서 기다림 |
| **🔔 디스패치 (Dispatch)** | "차례 되셨습니다! 3번 테이블로!" (전이 과정) |
| **실행 (Running)** | 테이블에서 밥 먹는 중 |
| **대기 (Waiting/Blocked)** | 주문한 음식 안 나와서 멍 (I/O 대기) |
| **종료 (Terminated)** | 계산하고 나감 |

**흐름**: 준비 → 실행 → 대기 → 준비 → 실행 → 종료

### IPC (Inter Process Communication) — 101호 도마 비유 ⭐

너 정답. 101호(배민)와 102호(토스)는 서로 못 들어감 → **경비실(OS)이 만든 공동 택배함(IPC)에 메시지 남김**.

**실제 예**: 배민에서 결제 누르면 토스 앱 켜지면서 금액 자동 입력 = IPC 작동.

### 스케줄링 — 3가지 기본 + 현대

| 방법 | 약자 | 방식 | 단점 |
|---|---|---|---|
| **FCFS** | First-Come First-Served | 선착순 (은행 번호표) | 앞 사람 오래 걸리면 뒤 대기 |
| **SJF** | Shortest Job First | 짧은 일 먼저 | 큰 일 영원히 밀림 (**기아**) |
| **라운드 로빈** | Round Robin | 2초씩 돌아가며 (공평) | - |

**현대 OS 표준**:
- Windows·macOS·Linux = **다단계 피드백 큐 (Multilevel Feedback Queue)**
- 라운드 로빈 + 우선순위 (사용자 활성 앱 우대)
- 스마트폰 = 전력 효율 코어 우선

### 교착상태 vs 기아상태

| | 교착상태 (Deadlock) | 기아상태 (Starvation) |
|---|---|---|
| **상황** | 서로 자원 안 놓고 대치 | 우선순위 낮아서 영원히 밀림 |
| **비유** | 외나무다리 마주침 | 단골손님에 밀려 굶음 |
| **FCFS/RR에서?** | 발생 가능 | **발생 ❌** (모두 차례 옴) |
| **SJF/우선순위에서?** | 발생 가능 | **발생** |

## 🙋 은미 정수 3 (2026-05-29) — 메모리 관리 깊이 (Day 2 02강)

### 단편화 (내부 vs 외부) — 식당 좌석 비유 ⭐

| | 상황 | 식당 |
|---|---|---|
| **내부 단편화** | 칸이 너무 커서 남는 공간 못 씀 | 4인석에 3명 → 1석 낭비 |
| **외부 단편화** | 빈 자리 흩어져서 큰 거 못 받음 | 6명 단체 왔는데 빈 자리 6개가 흩어짐 |

### 가상 메모리 3대 기능 ⭐ (안심번호 비유의 진짜 뜻)

| 기능 | 식당 비유 | 설명 |
|---|---|---|
| **용량 확장** | 식당 뒤편 마당에 대기석 | RAM 좁을 때 SSD를 메모리인 척 |
| **주소 변환 (보안)** | 손님에 가짜 테이블 번호 | 진짜 RAM 주소 숨김. **안심번호 비유 = 이거** |
| **메모리 공유** | 셀프바 (모두 같이 보는 반찬) | 공통 코드 1번만 올리고 여러 프로세스가 공유 |

**카톡 메모리 공유 예시**:
- 카톡 창 3개 띄워도 카톡 프로그램 자체는 RAM에 1개만
- 3개 창이 그걸 같이 봄
- **나만의 데이터 (대화 내용)** 만 따로 저장

### 페이지 vs 페이지 테이블 — 퍼즐 비유 ⭐

너 정답: **페이지 = 퍼즐 조각 / 페이지 테이블 = 퍼즐 위치 안내판**.

- **페이지** = 프로그램을 4KB로 잘게 찢은 조각 (외부 단편화 해결)
- **페이지 테이블** = "1번 조각은 RAM의 50번 자리" 같이 적어둔 장부

식당: "단체 손님 다 1명씩 찢어서 흩어진 자리에 앉히기" + "누가 어디 앉았는지 장부에 기록"

### 요구 페이징 (Demand Paging) + 쓰레싱 ⭐⭐

**요구 페이징** = 프로그램 전체 안 올리고 **지금 필요한 페이지만** RAM에 올림.
- 게임 10GB → 현재 맵 데이터만 RAM에
- 나머지는 SSD에 둠 → 다음 맵 갈 때 가져옴

**쓰레싱 (Thrashing)** — 렉의 진짜 정체:
- RAM 작은데 큰 프로그램 돌리면
- 1번 페이지 ↔ 2번 페이지 무한 교체
- CPU는 일 못 하고 페이지 갈아끼우는 데 시간 다 보냄
- → 우리가 일상에서 "컴퓨터 버벅거리는 거"의 정체

### 세그멘테이션 = 의미 단위 분할

페이지(기계적 4KB)와 다른 메모리 쪼개기 방법.

| | 페이지 | 세그멘테이션 |
|---|---|---|
| **기준** | 무조건 4KB | 의미 단위 (코드·데이터·함수) |
| **크기** | 일정 | 제각각 |
| **식당** | 손님 다 1명씩 찢기 | 가족은 4인석·커플은 2인석 |

→ 현대 OS는 둘 다 섞어 씀.

## 🙋 은미 정수 4 (2026-05-29) — 컴퓨터 하드웨어 정리

### CPU vs RAM vs SSD vs ROM — 식당 비유 완결

| 부품 | 식당 비유 | 역할 |
|---|---|---|
| **CPU** | 요리사 실력·손 속도 | 계산·실행 |
| **RAM** | 요리사 앞 조리대 | 지금 실행 중인 거 임시 |
| **SSD/HDD** | 식당 뒤편 창고 | 영구 저장 |
| **ROM (진짜)** | 식당 오픈 매뉴얼 (못 바꿈) | 부팅 명령어 (BIOS) |

### ⚠️ "스마트폰 ROM 256GB"의 실체

**스마트폰 ROM 256GB = 사실 SSD**. 마케팅 용어 오용. 진짜 ROM은 BIOS 같은 부팅 칩 (작고 못 바꿈).

### HDD vs SSD

| | HDD | SSD |
|---|---|---|
| **구조** | 자석 디스크 + 바늘 (물리 회전) | 반도체 칩 (전자) |
| **속도** | 느림 | **수십 배 빠름** |
| **소음** | 위잉 | 무소음 |
| **가격** | 쌈 | 비쌈 |
| **용도** | 백업·대용량 | 부팅·메인 |

**NVMe** = SSD 중에서도 초고속 (M.2 슬롯). 맥북 기본 = NVMe SSD.

### RAM 용량별 가이드 (2026)

| RAM | 식당 | 적정 프로그램 |
|---|---|---|
| 8GB | 분식집 | 웹서핑·문서·카톡 |
| **16GB** ⭐ | **표준 식당** | **일반 PC 게임·포토샵·코딩** ← 은미 맥북 |
| 32GB | 대형 뷔페 | 4K 영상·3D 렌더링 |
| 64GB+ | 급식소 | AI 학습·서버 |

→ 은미 16GB 맥북 = 코드 작업 충분. M 시리즈 칩 + 16GB = 현업 개발자 표준 구성.

### 개발 성능 결정 3대 요인

1. **CPU 종류** (Apple M3/M4 > Intel)
2. **RAM 용량** (16GB부터 쾌적)
3. **화면 크기** (외장 모니터 추가 시 효율 ↑↑)

## 🙋 은미 정수 5 (2026-05-29) — 파일 시스템 (Day 2 03강)

### 파일·디렉토리·메타데이터

| | 정체 | 비유 |
|---|---|---|
| **파일** | 알맹이 (내용물) | 상자 안의 물건 |
| **디렉토리(폴더)** | 파일 모은 방·상자 | 상자 |
| **메타데이터** | 파일의 견출지 (속성) | 크기·생성일·권한·위치 |

### 할당 4방식 — 디스크에 파일 배치

| 방식 | 비유 | 특징 |
|---|---|---|
| **연속** | 기차 이어붙이기 | 빠름·중간 삭제 시 빈 공간 |
| **연결** | 보물찾기 (다음 조각 따라감) | 빈 공간 ❌·중간 고장 위험 |
| **FAT** | 네비게이션·전체 안내 지도 | USB 표준·다음 조각 다 적힘 |
| **색인 (Index)** | 치트키 힌트 (모든 위치 한 장) | 직접 접근 빠름 |

### inode = 파일 주민등록번호 ⭐

- 리눅스/유닉스 표준
- 파일마다 고유 inode 번호
- inode 안에: **메타데이터 + 디스크 주소**
- 시스템은 이름 대신 inode 번호로 식별
- 확인: `ls -i`

### 심볼릭 링크 vs 하드 링크 ⭐⭐

| | 심볼릭 (Symlink) | 하드 링크 |
|---|---|---|
| **비유** | 윈도우 바로가기 | **하나의 영혼 + 두 개 육체** |
| **inode** | 다름 (새로) | **같음** |
| **원본 삭제** | 깨짐 (에러) | **다른 쪽 살아있으면 데이터 보존** |
| **명령어** | `ln -s 원본 새이름` | `ln 원본 새이름` |
| **드라이브 간** | OK | ❌ 같은 드라이브만 |
| **폴더** | OK | ❌ |

**하드 링크 사용 예**:
- 10GB 영화 → 2개 폴더에 하드 링크 → 용량 그대로 10GB
- 한쪽 지워도 데이터 보존

### 마운트 = 연결 선언 (복사 ❌)

리눅스는 C:·D: 드라이브 개념 ❌. 모든 게 **하나의 폴더 트리**.

```
새 USB 꽂음 → 빈 폴더(/mnt/usb) 선택 → "이 폴더 = USB 내부 인식"
```

→ 폴더 열면 USB 내부 보임. 복사 ❌. **연결 선언**.
