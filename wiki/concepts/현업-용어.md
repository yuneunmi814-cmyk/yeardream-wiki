# 현업 IT 용어 묶음

> **묶음 페이지 안내**: 「현업에서의 용어 노하우」(32p)는 5개 하위 주제가 한 자료에 묶여 있어 일단 묶음 페이지로 작성. 후속 강의에서 동일 개념이 다시 등장하면 별도 페이지로 분리·누적([[CLAUDE]] 절대 규칙 5).

## 한 줄 정의

현업 IT 환경에서 비전공자가 자주 마주치는 5대 핵심 용어·도구의 작동 원리: **Version Control / 기획·플로우차트 / 오픈소스 / 클라우드 / SaaS**.

---

## 1. Version Control (Git)

### 한 줄
프로젝트 코드의 변경 이력을 시점별로 관리하고, 협업·롤백·브랜칭을 가능하게 하는 도구.

### 핵심
- **Git이 사실상 표준** — 리누스 토발즈(리눅스 OS 제작자) 작품
- 대표 서비스 두 곳:
  - **GitHub** (Microsoft가 8조원에 인수)
  - **Bitbucket** (Atlassian — 공동창업자가 호주 부호 6위)
- "설탕 1·2·3스푼" 비유: 어느 시점으로든 이동 가능
- 협업·테스트·출시·제작 등 **버전 분리 작업** 용이
- Git 공식 자기 소개 (강의 인용): "free and open source distributed version control system" — 분산형, 빠른 성능, Subversion·CVS·Perforce·ClearCase 같은 SCM 도구 능가

### 실 예시 — 카카오톡 v9.9.0 업데이트 노트

```
• 오픈채팅 홈 아이콘 추가
• 내프로필 QR 개선
• 오픈채팅 송금 기능 추가
• 이미지 편집기 개선
• 보이스룸 공감 아이템 추가
• 이모티콘 즐겨찾기 개수 확대 (최대 120개)
```

→ 각 항목 = 한 버전에 들어간 변경사항. **출시 노트가 곧 Version Control의 사용자 측면 표현**.

### 왜 중요한가
사용자는 이미 Claude Code 일상 사용자 → Git 작동 지식은 있을 것. 강의에선 비전공자 입문 톤.

## 2. 기획과 플로우차트

### 한 줄
서비스의 핵심 흐름을 데이터와 함께 시각화한 도식. 기획-개발 간 공통 언어.

### 강의 예시 1 — 쇼핑몰 핵심 로직

```
[상품 페이지 접속] → [상품 구매] → [상품 배송]
                                ↑
                    배송 정보가 없네? → 로그인/회원가입 추가

[상품 페이지] → [로그인] → [구매] → [배송]
                  ↑
        이름·전화·주소 자연스럽게 획득
                            ↑
                 간편결제(카카오페이/네이버페이)·카드·무통장
                            ↓
                  배송사 API 레퍼런스 체크
```

### 강의 예시 2 — 미스터리 박스 (분기 플로우)

```
[결제] → [미스터리 박스 구매 완료] ──오픈──→ [상품 당첨] ─┬─→ [배송]
                  │                                      │
                  ↓                                      └─→ [환불] → [상품 가격 포인트 환불]
            [100% 환불]                                       ↓
                  │                                     [마이페이지]
                  └────── 마이페이지 정보저장 ──────→         ↑
                                                              마이페이지 정보저장
```

→ 분기(당첨/환불/100% 환불)가 모두 **마이페이지로 정보 저장 수렴**. 단일 흐름이 아닌 여러 경로 처리 패턴.

### 프로토타입 이후의 작업

- 기획팀의 **상세 기획** 시작
- 개발자는 **레퍼런스·라이브러리·API 등 점검** ([[라이브러리-vs-프레임워크]], [[REST-API]])

### 왜 중요한가
1인 컨설팅에서 클라이언트한테 "이렇게 동작합니다" 보여줄 때 그대로 사용 가능. **프로젝트윤365 방법론 자산**으로 풀 만한 1번 케이스.

### 자주 헷갈리는 것
- 와이어프레임(화면 단위) vs 플로우차트(흐름 단위) — 다른 도구

## 3. 오픈소스

### 한 줄
프로젝트 전체 코드가 공개되어 누구나 활용·기여할 수 있는 소프트웨어. 단 **무료가 곧 자유는 아님** — 라이선스로 관리됨.

### 핵심
- **OLIS** (오픈소스 SW 라이선스 종합정보 시스템): https://www.olis.or.kr/license/compareGuide.do
- 안드로이드도 오픈소스 OS
- 대표 오픈소스: Firefox, OpenOffice, Blender, PHP, Java, MySQL, Debian, WordPress, Apache, Ubuntu, Linux
- 어플리케이션 메뉴에 "사용 중인 오픈소스 SW" 명시 = 라이선스 규칙 준수
- 대표 라이선스 카테고리:
  - **GPL** (전염성 강함, 파생물도 공개 의무)
  - **MIT / Apache** (상대적으로 자유)

### 실 예시 — 카카오톡 OSS Notice

라이선스 규칙에 따라 카카오톡이 자사가 사용하는 오픈소스를 안드로이드 메뉴에 기재:

```
• abseil-cpp (Apache License 2.0) — github.com/abseil/abseil-cpp
• android-platform-cts (Apache License 2.0)
• android-platform-frameworks-base (Apache License 2.0)
• android-platform-frameworks-ex (Apache License 2.0)
• android-platform-frameworks-multidex (Apache License 2.0)
• android-platform-frameworks-opt-calendar
• ... (계속)
```

→ **사용자가 1인 컨설팅에서 클라이언트 앱 빌드 시 동일한 OSS Notice 작성 의무** 발생. Apache License 2.0이 대세.

### 왜 중요한가
**프로젝트윤365 클라이언트 솔루션 빌드 시 직접 영향**. GPL 코드 사용하면 클라이언트 코드도 공개해야 하는 함정. 콘텐츠 후보 1번 → "오픈소스 라이선스 모르고 쓰면 큰일나는 케이스 3가지".

## 4. 클라우드

### 한 줄
실물 서버 없이 인터넷을 통해 서버·소프트웨어를 빌려 쓰는 모델. 사용한 만큼 과금, 유연한 확장.

### 핵심
- **기존 서버의 단점**: 용량 확장 경직, 유지비 고정, 계산 틀리면 추가비용 부담
- **클라우드의 장점**:
  - 유연한 셋업
  - **사용량 기반 과금** (pay-as-you-go)
  - 365일 정상 가동 운영 부담 감소
- 트래픽 증감 → 증설·폐기 클릭 몇 번 (강의 명시 시나리오: 증가 → 증설 → 감소 → 폐기)
- **강의 명시 대표 서비스 3대**:
  - **AWS** (Amazon Web Services)
  - **Microsoft Azure**
  - **Google Cloud**
- (+ Cloudflare 등 사용자가 이미 쓰고 있는 서비스)

### 컨테이너화 (2일차 [[운영체제]] 마지막 슬라이드)

> "실행에 필요한 모든 환경과 패키지를 포함하여 하나로 묶는 기술. 내가 사용하는 OS에 상관없이 동일한 개발환경을 구성할 수 있음."

- Docker 추정. 클라우드 워크로드의 표준
- **1인 컨설팅 직결**: 클라이언트마다 다른 환경 차이로 인한 디버깅 시간 절감
- 자세히는 [[운영체제]] 컨테이너화 섹션 참조 (강의에선 한 슬라이드로 짧게)

### 왜 중요한가
사용자는 이미 **Cloudflare Pages** 사용자 → 클라우드 작동 지식 있음. 강의는 입문 톤일 듯. 1인 사업자 인프라 의사결정 멘탈 모델로 콘텐츠화 가능.

---

## 5. SaaS (Software as a Service)

### 한 줄
클라우드 기반의 구독형 소프트웨어 서비스. 설치·운영 부담 없이 브라우저·앱으로 바로 사용.

### 핵심
- 파일 공유 등 **공유의 유연성**
- 설치 과정 없이 **즉시 사용**
- **Adobe→Figma 28조원 인수** = SaaS 위력 사례
- 지속 확장 중인 핵심 시장

### 왜 중요한가
프로젝트윤365 자체가 잠재적 SaaS 비즈니스 후보. 인스타 콘텐츠 1번 후보 → "Figma는 왜 28조에 팔렸나 — SaaS 한 줄 설명".

---

## 자주 헷갈리는 것 (전체)

- **클라우드 vs SaaS**: 클라우드는 인프라 모델, SaaS는 그 위에서 돌아가는 소프트웨어 형태
- **IaaS / PaaS / SaaS** 3단계 구분 (강의 보충 후 갱신)
