# Library vs Framework

## 한 줄 정의 (강의 본문)

- **라이브러리**: 반복해서 자주 사용하는 **코드 모음 / 도구 모음** (= package)
- **프레임워크**: 프로그램을 효율적으로 짜기 위한 **틀(Frame)** — 코드 작성 구조 제시

## 핵심 아이디어

### 등장 배경 — 웹의 복잡화

```
[다양한 웹 브라우저들]  ⇄  HTML로 정보 주고받음  ⇄  [웹서버]
(Chrome, Firefox, Brave 등)                      (서버 컴퓨터)
```

#### 팀 버너스리 (Tim Berners-Lee) — www의 창시자

- 영국 물리학자
- 세계 최초의 웹사이트: **info.cern.ch** ([http://info.cern.ch/hypertext/WWW/TheProject.html](http://info.cern.ch/hypertext/WWW/TheProject.html))
- **HTML, CSS, HTTP 통신, URL 등 현대 웹의 초석**
- 연구자 자료 공유용으로 만든 웹을 **대중에게 무료로 공개** → 인터넷 보편화의 결정적 결정
- 기사작위·서울 평화상 수상

#### 웹의 복잡화 — 1998 vs 2008 네이버

- **1998.12.12**: 초기 검색 중심의 단순 레이아웃
- **2008.12.12**: 뉴스·카페·블로그·이미지 등 **복합 포털**로 진화

→ 결과:
- 반복적으로 자주 사용되는 코드들 등장
- 협업 프로젝트 증가
- **구조화된 체계의 필요성** = 라이브러리·프레임워크 등장 배경

### 라이브러리 — 강의 정의

> **반복해서 자주 사용하는 코드 모음 / 일종의 코드로 이루어진 도구 모음**

- 코드의 역사가 쌓이며 자주 쓰이는 기능이 발생
- 자주 쓰이는 기능을 **함수로 묶어 선언하여 사용**
- **오픈소스 영역에서 활발히 생산·공유**
- 다른 이름: **package**

#### 예시 1 — Chart.js (시각화)

```javascript
const config = {
  type: 'bar',
  data: data,
  options: {
    scales: { y: { beginAtZero: true } }
  }
};
const data = {
  labels: labels,
  datasets: [{
    label: 'My First Dataset',
    data: [65, 59, 80, 81, 56, 55, 40],
    backgroundColor: [
      'rgba(255, 99, 132, 0.2)',
      ...
    ]
  }]
};
```

→ "색과 데이터 수치만 입력한 간단한 선언으로, 복잡한 Chart를 쉽게 그려낼 수 있다" (강의 본문)

#### 예시 2 — Passport.js (인증)

- "Simple, unobtrusive authentication for Node.js"
- 로그인·회원가입 구현을 쉽게 도와줌
- **소셜 로그인** (카카오·네이버 포함) 쉽게 구현 가능
- Express 기반 웹 앱에 unobtrusively 드롭인 가능

### 라이브러리 사용 시 주의

- 검증된 라이브러리를 잘 활용하는 것도 **개발자의 능력**
- 데이터 처리 관련은 따로 학습이 필요할 정도로 큰 라이브러리도 존재
- **적절하지 못한 사용으로 성능 저하** 발생 가능

> **한 문장 요약**: 프로젝트의 생산성 향상과 효율성 증진을 위해 라이브러리를 활용한다.

### 프레임워크 — 강의 정의 + 빵틀 비유 ⭐

> **Frame = 어떤 형태의 틀을 가진 형태**
>
> **빵 모양을 만들기 위한 빵틀처럼, 코드를 작성하기 위한 구조를 제시**

특징:
- **공통된 규칙**을 활용해야 하므로 **협업에 매우 유리**
- 개발자는 **코드 로직 자체에 집중** 가능
- 언어별로 다양한 프레임워크 존재, 제공 도구가 다름

#### 예시 — Django (Python 웹 프레임워크)

- 파이썬 기반 웹페이지 작성용
- 보통은 직접 만들어야 하는 **관리자 페이지를 Django가 상당한 수준으로 제공**
- 제작 부담을 덜어줌

## 라이브러리 vs API (강의 명시 비교)

강의에 벤다이어그램으로 표현됨:

| 구분 | 공통점 | 차이점 |
|------|--------|--------|
| 라이브러리 | 재사용 가능한 코드의 반복 활용 | **내 컴퓨터(같은 프로세스) 안에서 호출** |
| API | 재사용 가능한 코드의 반복 활용 | **네트워크·통신을 통해 기능 제공** |

→ 사용자가 Meta 광고 API 사용 경험 → "API는 인터넷 너머의 라이브러리"로 매핑하면 직관적.

## ⚠️ 강의 본문 보완 — IoC / Hollywood Principle

**강의 본문엔 IoC(Inversion of Control)·Hollywood Principle 용어가 등장하지 않음**. 강의의 라이브러리 vs 프레임워크 구분은 "도구 모음 vs 구조를 제시하는 틀(빵틀)"에 그침.

**일반 보강** (CS 표준 표현):
- **IoC (Inversion of Control, 제어의 역전)**: 라이브러리는 내 코드가 부르고, 프레임워크는 프레임워크가 내 코드를 부른다 — 호출 방향이 역전됨
- **Hollywood Principle**: "Don't call us, we'll call you" — 프레임워크의 본질을 한 줄로 표현
- 빵틀 비유와 정확히 동일한 의미. 다만 "누가 누구를 부르냐"라는 판별 기준이 더 명확함

→ 강의에선 빵틀 비유로 충분하지만, **AI Agent 개발(LangChain·LlamaIndex 등)에서 프레임워크/라이브러리 선택 시 IoC 관점이 의사결정에 더 유용**.

## 왜 중요한가

- 사용자는 이미 **SwiftUI**(프레임워크)·다양한 npm 라이브러리(라이브러리)를 다 써본 상태 → 직관으로 알던 차이를 **명시적 멘탈 모델**로 정리
- 클라이언트 솔루션 설계 시 "프레임워크 채택 = 자유도 포기, 속도 획득" 트레이드오프 의사결정에 직결
- AI Agent 개발에서도 **LangChain(프레임워크) vs OpenAI SDK(라이브러리)** 식의 선택 구도 동일

## 자주 헷갈리는 것

- **React = 라이브러리? 프레임워크?**: 공식은 라이브러리. 단 Next.js·Remix 같은 메타프레임워크 포함하면 프레임워크급 구속력
- **"이거 라이브러리야 프레임워크야?"** 질문보다 **"이게 내 코드를 호출하나, 내가 이걸 호출하나?"** 로 판별 (= 빵틀 vs 도구)
- **라이브러리 vs API**: 위 비교표 참조. 핵심은 **네트워크 너머인지 여부**.
- **package = 라이브러리?**: 강의 본문에서 둘을 동의어로 사용. 실무에서도 거의 동의어 (npm package = JavaScript 라이브러리)
