# 해시 (Hash)

## 한 줄 정의

**임의 길이 데이터 → 고정 길이 출력**. 단방향 (복호화 불가). 패스워드 저장·무결성 검증·전자서명에 필수.

## 핵심 아이디어

```
"Hello"      → SHA-256 → 185f8db32271fe25f561a6fc938b2e264306ec304eda518007d1764826381969
"Hello!"     → SHA-256 → 334d016f755cd6dc58c53a86e183882f8ec14f52fb05345887c8a5edd42c87b7
"안녕하세요" → SHA-256 → b15...
```

특징:
- **고정 길이** (입력 길이 무관, 출력 항상 같은 길이)
- **단방향** — 결과로 입력 못 찾음
- **민감성** — 1글자 변경 시 완전히 다른 해시 (Avalanche 효과)
- **무충돌** (이론상, 같은 해시 갖는 다른 입력 찾기 어려움)

### 알고리즘별 안전성

| 알고리즘 | 출력 | 안전성 | 실사용 |
|---|---|---|---|
| **MD5** | 128 bit | ❌ 깨짐 | 무결성만 (보안 ❌) |
| **SHA-1** | 160 bit | ❌ 깨짐 (2017) | 레거시만 |
| **SHA-256** | 256 bit | ✅ 안전 | **현재 표준** |
| **SHA-512** | 512 bit | ✅ 안전 | 고보안 |
| **SHA-3** | 가변 | ✅ 최신 | 미래 |

### 3대 활용

1. **패스워드 저장**
   ```
   DB: { user: "yoon", password_hash: "5e88..." }
   ```
   유출돼도 원본 모름. 로그인 시 입력 해시 → DB 해시 비교.
   (실무는 솔트·반복 해싱 추가 = `bcrypt`·`scrypt`·`Argon2`)

2. **파일 무결성 검증**
   ```
   downloaded.zip의 해시 == 공식 페이지 해시 → 변조 없음
   ```

3. **전자서명** — Private Key로 메시지 해시 암호화

### 역상 저항성

- **제1 역상 저항성**: 해시값으로 입력 찾기 어려움
- **제2 역상 저항성**: 같은 해시 갖는 다른 입력 찾기 어려움

## 왜 중요한가

- **모든 인증 시스템 기반** — 패스워드·[[JWT]] 서명·[[SSL-TLS]] 인증서
- **블록체인 핵심** — Bitcoin·Ethereum 다 SHA-256 사용
- **현대 보안의 기둥**

## 자주 헷갈리는 것

### "해시 = 암호화"

다름. **암호화는 복호화 가능**, **해시는 불가능**. 그래서 패스워드 저장에 해시가 적합 (서버도 원래 비밀번호 몰라야 함).

### "해시는 안전하니까 그냥 쓰면 됨"

❌. **레인보우 테이블** 공격 있음 (자주 쓰는 비밀번호 해시 미리 계산). → **솔트(Salt)** 추가 + 반복 해싱 필요 (bcrypt 등).

### "MD5 아직 써도 됨"

❌ 보안 용도엔 절대 X. 파일 체크섬 정도만 OK.

## 일상 비유: 종이를 갈아 가루로

- 종이 → 분쇄기 → **가루** (해시)
- 어떤 종이든 같은 양의 가루 → **고정 길이**
- 가루 보고 원래 글자 못 알아냄 → **단방향**
- 종이 한 글자만 달라도 가루 모양 완전 달라짐 → **민감성**
