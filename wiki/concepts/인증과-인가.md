# 인증과 인가 (Authentication vs Authorization)

## 한 줄 정의

**인증**(Authentication) = "**너 누구야?**" / **인가**(Authorization) = "**그거 해도 돼?**". 보안의 가장 기본 두 단계.

## 핵심 아이디어

| | 인증 (Authentication) | 인가 (Authorization) |
|---|---|---|
| **질문** | "너 누구야?" | "그거 해도 돼?" |
| **예** | 로그인 (ID/PW) | 관리자 페이지 접근 권한 |
| **순서** | **먼저** | 인증 후 |
| **결과** | 토큰·세션 ID 발급 | 자원 접근 허용/거부 |
| **줄임** | AuthN | AuthZ |
| **HTTP 코드** | 401 Unauthorized | 403 Forbidden |

### 순서가 중요

```
정문 (인증) 통과 못 하면 → 9층 임원실 권한 따질 일도 없음

[로그인 시도] → [인증] → 실패 → 401
                  ↓ 성공
              [인가] → 권한 부족 → 403
                  ↓ 성공
              [자원 접근]
```

### 인증 요소 4종

1. **Knowledge (아는 것)** — 비밀번호·PIN
2. **Possession (가진 것)** — OTP·인증서·USB 키
3. **Inherence (본인인 것)** — Face ID·지문·홍채
4. **2FA·MFA** — 위 중 2개 이상 조합

## 왜 중요한가

- **모든 보안의 출발점** — 모든 인증 메커니즘([[JWT]]·[[쿠키]]·[[세션]])이 둘 중 하나로 분류
- **HTTP 401 vs 403** 의 의미 — 디버깅 시 핵심
- **서비스 설계** — 사용자 권한 모델 결정

## 자주 헷갈리는 것

### 401 vs 403

- **401 Unauthorized**: 인증 실패. "**너 누구야?**" (로그인 안 됨)
- **403 Forbidden**: 인가 실패. "**누군지는 알겠는데 안 됨**" (권한 부족)

이름이 헷갈리게 지어졌지만, 코드 의미는 분명.

### "인증 = 인가"

전혀 다른 단계. **인증은 신원**, **인가는 권한**.

## 일상 비유: 회사 출입 절차

- **인증** = 정문 사원증 확인 ("너 직원 맞아?")
- **인가** = 9층 임원실 카드키 ("**임원만** 들어와")

직원이라는 건 인증, 임원이라는 건 인가. 인증 통과 못 하면 인가는 따질 필요도 없음.
