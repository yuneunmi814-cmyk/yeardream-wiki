# Indexing (색인)

## 한 줄 정의

많은 데이터가 쌓여 검색이 느려질 때, **검색 속도를 높이기 위해 데이터를 저장할 때 별도의 구조를 추가로 저장**하는 방법. 검색엔진에선 "수집한 웹페이지를 검색 가능한 형태로 정리"하는 단계.

## 핵심 아이디어

### 구글 공식 정의 (강의 인용)

> "Google에서는 알고 있는 모든 웹페이지를 색인에 저장합니다. 각 페이지의 색인 항목은 해당 페이지의 콘텐츠 및 위치(URL)를 명시합니다. Google에서 페이지를 가져와서 읽고 색인에 추가하면 색인이 생성됩니다."

→ 색인 항목 = `(콘텐츠, URL)` 쌍

### 검색엔진 3단계에서의 위치

```
크롤링 → [색인 생성] → 검색결과 제공
```

[[크롤링]]으로 가져온 페이지를 **검색 가능한 자료구조**(예: 역색인, inverted index)로 정리해서 저장. 이 작업이 없으면 검색마다 모든 페이지를 다 훑어야 함.

### SEO와의 직접 연결

> "색인 과정에서 더 효율적인 노출이 될 수 있도록 작업하는 것 = SEO 작업의 핵심" (강의 원문)

[[SEO]] 작업의 본질은 **색인되는 방식·색인 안에서 노출되는 우선순위에 영향을 미치는 것**. 결국 SEO는 indexing 단계를 타겟팅.

## 실습 트릭 — `site:` 검색으로 색인 확인

```
구글 검색창 → site:naver.com
```

→ 구글이 색인한 naver.com 하위 페이지 목록 출력 (네이버 고객센터, 네이버 부동산, OGQ마켓, 네이버 클로바 등).

본인 도메인(예: 향후 yoon.ai.lab 랜딩)에 적용하면 **어떤 페이지가 색인됐고 어떤 게 누락됐는지** 즉시 확인 가능.

## 왜 중요한가

- **SEO의 직접 작업 대상** — 색인 안 되면 검색결과에 등장할 자격조차 없음
- **검색 성능의 토대** — 데이터베이스의 인덱스도 같은 원리, [[REST-API]] 백엔드 설계 시에도 등장
- **사용자 채널 운영**: yoon.ai.lab / prompt.by.yoon 외부 랜딩·블로그 만들 때 "색인됐는지" 확인이 SEO의 첫 체크포인트

## 자주 헷갈리는 것

- **검색엔진 인덱스 vs DB 인덱스**: 원리는 같음(검색 속도 향상용 별도 자료구조). 검색엔진은 웹페이지 단위, DB는 테이블 컬럼 단위. 강의에선 둘을 굳이 구분하지 않음.
- **색인 vs 랭킹**: 색인은 "결과 후보에 들어갈 자격", 랭킹은 "그 안에서 몇 번째로 노출될지". SEO는 둘 다 영향.
- **색인 등록 ≠ 즉시 노출**: 사이트맵 제출·등록 도구(네이버 Search Advisor, 구글 Search Console) 사용해도 색인 반영까지 시간 걸림.

## 🙋 은미 정수 (2026-05-29) — 색인 정확히

**은미 직감 ✅ 절반 정답**: 해시태그도 색인의 일종.

| | 해시태그 #한식 | 검색엔진 색인 |
|---|---|---|
| **만드는 주체** | **사용자가 직접** | **구글이 자동** |
| **방식** | "내가 이 게시물은 '한식'" | 페이지 자동 분석해 단어 추출 |
| **비유** | 본인 라벨링 책꽂이 | 사서 자동 분류 도서관 |

→ **둘 다 색인**. 단 **자동(구글) vs 수동(해시태그)**.

→ 강의에서 "색인"이라 하면 보통 **구글 자동 색인**.
