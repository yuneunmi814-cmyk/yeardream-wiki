# App / Web Services

## 한 줄 정의

사용자에게 도달하는 앱·웹 서비스가 어떻게 구성되는지 — 클라이언트(프론트엔드) ↔ 웹 서버 ↔ 백엔드 ↔ 데이터베이스로 이어지는 통신과 책임 구조.

## 핵심 아이디어

### 강의 시리즈 러닝 예제 — 디지털 명함 앱

```
명함 촬영 → 명함 사진 정보 전송 → DB 직접 입력 → 디지털 명함
```

→ 시리즈 전체에서 이 명함 앱이 일관된 학습 예시로 사용됨.

### Web Application Server (WAS) 기본 구조

```
[클라이언트]   →요청→   [Web Server]  ⇄  [백엔드]  ⇄  [DB]
(웹 브라우저,  ←응답←   (http 통신)        동작 코드    (각종 정보 저장)
 스마트폰)
```

→ 자세한 통신 흐름은 [[HTTP-프로토콜]], DB 부분은 [[데이터베이스-기초]] 참조.

### Endpoint

> "웹 서버의 주소를 가리키는 문자" (강의 본문)

브라우저에 입력하는 `www.google.com`, `www.naver.com` 같은 URL이 endpoint. → [[REST-API]]의 핵심 단위.

### Front-end (클라이언트)

> "서비스에서 처음 만나는 화면으로, 서비스의 의도대로 사용자를 유도"

주요 책임 4가지:
1. **UI/UX** — 화면과 사용자 경험
2. **독립적 기능** — 서버 무관하게 동작 (예: 카메라 촬영)
3. **서버 통신** — HTTP 요청
4. **사용자 패턴 분석** — Analytics

### UI vs UX

- **UI (User Interface)**: 사용자가 만나는 화면, 디자인 요소
- **UX (User Experience)**: 서비스 성격에 따라 사용자에게 편리한 경험 제공

> **GUI 메모**: 현대엔 그래픽 환경이 압도적이라 GUI/UI 구분 없이 "UI"로 통용 (강의 본문 메모)

UX 예시 — 카메라 촬영 도달 경로:
- 긴 경로: 로그인 → 메인 → 메뉴 → 기능 → 촬영
- 짧은 경로: 로그인 → 메인 하단 버튼 → 촬영

### 반응형 화면 & Grid System

- **Grid System** (웹): 수학적 논리로 페이지 콘텐츠를 일관 구조화 (2/3/4/6 Columns 분할)
- **Component System** (모바일): 동일 개념의 모바일 버전
- 다양한 스크린 사이즈 커버 + **동일 정보 제공**

### Back-end

> "서버 측의 동작 Code 비즈니스 로직(Business logic)을 처리" (강의 본문)

> **비즈니스 로직 메모**: 컴퓨터 프로그램에서의 데이터를 **생성·표시·저장·변경**하는 모든 내용

활용 예시 (강의 본문):
- 명함 정보 저장·변경
- 쇼핑몰 상품 구매·결제
- SNS 회원 가입

### 디지털 명함 통신 흐름 (전체 그림)

```
[명함촬영]
  → 명함정보 저장 요청 (POST)
                ↓
        [Web Server (HTTP 통신)]
                ↓
        [Hypertext 파싱 후 저장할 공간 찾기]  ← [[파싱]]
                ↓
        [DB에 정보 저장]                      ← [[데이터베이스-기초]]
                ↓
  ← 명함정보 저장 여부
[명함촬영]
```

### REST API 첫 등장 (자세히는 [[REST-API]])

> "약속된 지정된 주소(URI)를 가지고 있고, 각 주소에 기능(함수)을 가지고 있는 형태" (강의 본문)

예시 URL: `https://school.elice.io/courses/26028/info` (본 강의 URL)

> **URI vs URL 메모**: URI는 서버에서의 특정 리소스 식별자, URL은 흔히 사용하는 웹사이트 주소. **URI가 URL보다 더 큰 의미**라는 것만 기억하고 넘어가자 (강의 본문).
> 
> 보강: 정확히는 **URI = URL ∪ URN**. URN은 위치와 무관한 이름(예: `urn:isbn:0451450523`). 실무에선 URL과 URI를 거의 동의어로 사용.

## 왜 중요한가

- Stage 3 AI 서비스개발자 트랙(Option B)의 토대
- 프로젝트 산정 시 "이 구조면 이만큼 시간·비용" 멘탈 모델
- 사용자는 이미 SwiftUI(클라이언트)·Cloudflare Pages(정적 호스팅) 경험 → **자기 경험을 강의 용어와 매칭**하는 단계가 학습 효과 큼

## 자주 헷갈리는 것

- **Web Server vs Back-end**: 강의에선 분리해서 그림. Web Server = HTTP 요청 받는 부분, Back-end = 비즈니스 로직 처리. 실무에선 Express·Django 같은 프레임워크가 둘을 한 번에 담당해서 경계가 흐려짐.
- **앱 vs 웹**: 네이티브 앱(iOS·Android)·하이브리드(React Native)·웹앱(PWA)·일반 웹 — 모두 클라이언트지만 동작 방식 다름.
- **Endpoint = URL?**: 강의에선 거의 같은 의미로 쓰지만, 엄밀히는 endpoint = API의 특정 기능을 가리키는 URL.

## 🙋 은미 정수 (2026-05-29) — 막혔다가 푼 것

### 앱 / 웹 / 서비스 구분 — 인스타 기준

| | 뜻 | 인스타 예 |
|---|---|---|
| **앱** | 휴대폰·PC 설치하는 것 | iOS·Android 인스타 앱 |
| **웹** | 브라우저로 보는 것 (URL) | instagram.com |
| **서비스** | 앱·웹 + 뒤의 서버·결제·계정 전부 | 인스타 = 모바일 앱 + 웹사이트 + 백엔드 서버 + Meta 계정 시스템 + 광고 시스템 |

→ "**서비스가 가장 큰 단어**". 앱·웹은 그 일부.

### WAS vs 웹 서버 — 인스타 비유 ⭐

| | 웹 서버 (정적) | WAS (동적) |
|---|---|---|
| **방식** | 미리 만든 HTML 그대로 | 그때그때 사용자별로 만들어서 |
| **예** | 회사 소개 페이지 (모두 같음) | 인스타 피드 (사람마다 다름) |
| **마인크래프트 비유** | 게임 매뉴얼 PDF (다 같음) | 멀티 서버 (내 인벤토리·위치 실시간 응답) |

→ 인스타·유튜브·카톡·게임 멀티 서버 = 다 **WAS**.

### 기술 스택 도구 — 발음 + 풀이

**정적 웹 서버**:
- **Nginx (엔진엑스)** — 2004 러시아. 가볍고 빠름. 인스타·넷플릭스 다 씀. 정적 파일·리버스 프록시 최강
- **Apache (아파치)** — 1995. 가장 오래된 정적 서버. 무거움·유연함. LAMP 스택의 A

**동적 WAS (백엔드 프레임워크)**:
- **Node.js (노드 / 노드제이에스)** — 2009. 자바스크립트를 서버에도. 실시간 채팅 강함. 비동기 잘함
- **Express (익스프레스)** — Node.js 위에 올리는 프레임워크. 항상 짝: **Node.js + Express**
- **Java + Spring (자바 + 스프링)** — 한국 대기업·은행·정부 99%. 안정·대규모. 1995/2003
- **Ruby on Rails (루비 온 레일즈)** — 2004. "관례 over 설정" 빠른 개발. Twitter·Shopify 초기
- **Python + Django (파이썬 + 장고)** — 2005. 전통 풀스택. 인스타도 Django
- **Python + FastAPI (파이썬 + 패스트에이피아이)** ⭐ — 2018, **AI 시대 표준**. Stage 3 가면 거의 이거

**왜 기술 스택이 묶이는가?** (3가지 이유):
1. **언어 호환성** — 같은 언어로 묶으면 개발자 이동·코드 공유 쉬움
2. **목적 매칭** — 빠른 개발 → Rails / 안정 → Spring
3. **커뮤니티 크기** — 사람 많은 스택 = 자료·도구·튜토리얼 풍부

**대표 스택**:
- **MERN** = MongoDB + Express + React + Node.js (다 JavaScript, JS 풀스택)
- **LAMP** = Linux + Apache + MySQL + PHP (다 무료, 비용 ↓)
- **Python + FastAPI + PostgreSQL + React** ⭐ — **AI 시대 표준**

### FastAPI 더 깊이 (2026-05-29 추가)

**FastAPI = Python으로 API 서버 만드는 프레임워크**.

- 2018 등장, 빠름, 자동 문서 생성 (Swagger UI)
- AI 시대 표준 자리 잡음 (AI 모델 = Python → API도 Python)
- **비유**: Python = 주방 / FastAPI = 그 위 레스토랑 운영 시스템 (주문·계산·메뉴판)

실무 코드:
```python
from fastapi import FastAPI
app = FastAPI()

@app.get("/users/{user_id}")     # ← Endpoint
def read_user(user_id: int):
    return {"user_id": user_id, "name": "은미"}
```

→ 실행하면 즉시 API 서버 동작.

### LAMP 스택 풀이 (2026-05-29 추가)

옛 인터넷(2000~2010s)의 절반이 LAMP. **다 무료·오픈소스** → 비용 ↓:

| 글자 | 정체 | 역할 |
|---|---|---|
| **L**inux | OS | 서버 운영체제 |
| **A**pache | 웹 서버 | 정적 파일 |
| **M**ySQL | DB | 데이터 저장 |
| **P**HP | 언어 | 백엔드 로직 |

**PHP (피에이치피)** = 옛 웹 전용 언어 (1995). 워드프레스 = PHP. 한국 옛 게시판 다 PHP. 현재 인기 ↓ (Python·Node.js에 밀림).

→ 지금 새로 만든다면 LAMP보단 **Python + FastAPI + PostgreSQL + React**.

### 풀스택 정의 (2026-05-29 추가)

**Stack** = 쌓아 올린 기술 묶음. **Full Stack** = 다 쌓음.

| | 프론트엔드 | 백엔드 |
|---|---|---|
| **무엇** | 사용자가 **보는** 화면 | 사용자가 **못 보는** 서버 |
| **인스타 예** | 화면·버튼·애니메이션·피드 UI | 게시물 저장·DM 전달·좋아요 카운트 |
| **기술** | SwiftUI·React·Vue·HTML/CSS | Python·Node.js·Java·DB |

**풀스택 = 둘 다 만들 수 있는 개발자**. 혼자서 앱 처음부터 끝까지 가능.

### 취업 수요 (한국 2026 기준)

| 직무 | 수요 | 연봉 | 진입 |
|---|---|---|---|
| **프론트엔드** | 많음 | 중 | 중 |
| **백엔드** | 매우 많음 (대기업·금융) | 중상 | 중상 |
| **풀스택** | 스타트업 강함 / 대기업 분업 | 중상 | 높음 |
| **AI 엔지니어** | ⭐ 급증 | 상 | 상 |
| **모바일 (iOS/Android)** | 안정적 | 중상 | 중상 |

**트렌드**: 대기업 분업 / 스타트업 풀스택 선호 / **AI 엔지니어 최고 수요**.

**은미 매핑** (참고):
- 학습 목표 = **풀스택 + AI Agent 개발자** 조합 가까움
- SwiftUI(프론트) ✅ 절반 도달 + Python 익히면 풀스택 진입

### 백엔드 보조 언어 — Go · Rust (2026-05-29 추가)

**둘 다 백엔드용 빠른 언어**. ChatGPT 같은 대규모 서비스에서 Python과 함께 보조로 씀.

| 언어 | 발음 | 특징 | 어디 |
|---|---|---|---|
| **Go** | 고 | Google 2009. 빠르고 간단, 동시 요청 처리 강함 | Docker·Kubernetes·Uber·Cloudflare |
| **Rust** | 러스트 | Mozilla 2010. C++만큼 빠르지만 안전. 어려움 | Discord·Dropbox |

→ **은미는 굳이 안 배워도 됨**. AI 서비스 만들 때 보조 도구로 알고만 있으면 됨.

### TypeScript (타입스크립트) — 프론트엔드 표준 (2026-05-29 추가)

**TypeScript = JavaScript에 "타입" 추가한 언어**. 2012 Microsoft.

```javascript
// JavaScript: 타입 없음, 사고 가능
let age = 25
age = "스물다섯"   // OK, 나중에 사고

// TypeScript: 타입 명시, 에러 미리 잡음
let age: number = 25
age = "스물다섯"   // 컴파일러가 에러로 잡음!
```

→ Swift도 정적 타입이라 은미는 익숙할 수 있음. **현재 신규 프론트엔드 90% TypeScript**.

### AI 엔지니어 직무 깊이 (2026-05-29 추가)

**AI 엔지니어 = AI 모델 만들고 배포·관리**.

**세분화**:
- ML 엔지니어: 모델 학습·튜닝
- MLOps: 모델 배포·모니터링·자동화
- **AI Agent 엔지니어** ⭐ — 은미 매핑 (LLM 자동화 워크플로우)
- 데이터 엔지니어: 데이터 파이프라인

**일하는 흐름 (챗봇 만들기 예)**:
1. 요구사항 정의
2. 데이터 수집 / LLM API 선택
3. 모델 학습(PyTorch) 또는 Claude API 사용
4. 평가 (정확도)
5. 배포 (FastAPI + Docker)
6. 모니터링
7. 개선 반복

**도구**: Python + PyTorch + LangChain + Claude API + Docker + AWS

**은미 위치**: Claude Code 일상 사용 = AI Agent 도구 활용 경험 있음. Stage 3 트랙 결정 시 강력한 가산점.

**연봉·수요** (2026):
- 한국 신입 5,000만 / 경력 1억+
- 미국 신입 $200k / 경력 $400k+ (실리콘밸리)
