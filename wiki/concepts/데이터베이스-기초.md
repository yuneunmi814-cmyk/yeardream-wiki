# Database Basics

## 한 줄 정의

> "기록하고 조합하는 모든 데이터들의 총합" (강의 본문)

데이터를 **분류하기 좋게 구조화**하여 저장·검색·수정·삭제할 수 있게 만든 시스템.

## 핵심 아이디어

### 일상에서의 DB

- 통장의 입·출금 기록
- 도서관 도서 검색

은행앱 결제내역 조회 4단계:
1. 은행앱 열기
2. 은행 서버에 계좌정보로 접속
3. 접속 확인 / DB 검색
4. 결제 내역 확인

도서관 관리에 필요한 정보 항목:
- 책 보유 여부 / 책 정보(제목·작가·출판사) / 대여 중 여부 / 대여자 / 반납 예정일

→ **이런 항목들이 DB 컬럼이 됨**.

### CRUD — DB의 기본 4기능

```
[저장]  →  [검색]  →  [수정]  →  [삭제]
Create     Read       Update     Delete
```

→ HTTP 메서드와 1:1 매핑 ([[HTTP-프로토콜]] 참조).

### 수정 (Update) 예시 — 신발 매진 케이스

매진 컬럼 추가:

| 상품번호 | 상품종류 | 가격 | 매진 |
|---------|---------|------|------|
| 1 | 상의 | 50000 | False |
| 2 | 청바지 | 80000 | False |
| **3** | **신발** | **70000** | **True** |
| 4 | 벨트 | 10000 | False |

→ Boolean 컬럼으로 상품 라이프사이클 반영.

### 물리삭제 vs 논리삭제 ⭐

같은 매진 케이스, 두 가지 접근:

**물리삭제 (Hard Delete)**: 신발 row 자체를 DB에서 제거
```
1 | 상의 | 50000 | False
2 | 청바지 | 80000 | False
~~3 | 신발 | 70000 | True~~  ← 제거
4 | 벨트 | 10000 | False
```

**논리삭제 (Soft Delete)**: 데이터를 지우지 않고 플래그로 표시
```
1 | 상의 | 50000 | False
2 | 청바지 | 80000 | False
3 | 신발 | 70000 | True   ← 매진=True 유지
4 | 벨트 | 10000 | False
```

→ **실무 핵심 패턴**:
- 물리삭제: 진짜 필요 없는 데이터 (GDPR 개인정보 삭제 등)
- 논리삭제: 통계·복구·이력 추적이 필요한 경우 (대부분의 비즈니스 데이터)

콘텐츠 후보: "매진 신발로 보는 물리삭제 vs 논리삭제 1컷"

## DBMS와 DB 종류

### DBMS

대표 DBMS:

| DBMS | 제작사 |
|------|--------|
| MySQL | Oracle |
| MariaDB | MariaDB Foundation |
| PostgreSQL | PostgreSQL Global Dev Group |
| Oracle | Oracle |

### 관계형 (RDB) vs NoSQL

- **관계형 (RDB)**: **Primary Key**라는 고유값으로 테이블 간 연결
  - 예: "B신발을 구매한 A고객" → B신발 데이터 + A고객 데이터를 PK로 연결
  - 위 대표 DBMS 4종은 모두 RDB
- **NoSQL**: 기록 단순, **대용량 처리에 유리**
  - 예: MongoDB, Redis, Cassandra (강의엔 미언급, 일반 지식)

## 왜 중요한가

- **Stage 3 어느 트랙이든 일상 도구** — 분석가는 데이터 조회·정제, 서비스개발자는 설계·운영
- **1인 컨설팅 직접 영향** — 클라이언트 데이터를 어떻게 구조화할지가 솔루션 품질의 절반
- **AI 시대에도 DB는 안 없어짐** — RAG·벡터 DB도 결국 DB 패턴 + 의미 검색

## 자주 헷갈리는 것

- **DB vs DBMS**: DB는 데이터 자체, DBMS는 관리 소프트웨어. 실무에선 거의 혼용.
- **Primary Key vs Unique Key**: 둘 다 고유값이지만 PK는 테이블당 1개·NULL 불가, Unique는 여러 개·NULL 허용.
- **RDB vs NoSQL 선택**: "데이터 간 관계가 많으면 RDB, 양이 많고 구조가 단순하면 NoSQL"이 일반론. 실무는 둘 다 쓰는 게 흔함.
- **Query = SQL?**: SQL은 RDB용 Query 언어. NoSQL은 자체 Query 문법.

### 동시접속자 ↔ 서버 터짐 ↔ DB ⭐

**네 직감 ✅ 절반 정답**.

서버 터지는 원인 중 **DB 부하가 가장 흔함**:
- 한 사용자 요청 = DB 조회 여러 번
- 동시 1000명 = DB 쿼리 만 단위/초
- DB가 디스크 매번 접근 → 느림 → 병목

**다른 원인들**: CPU/메모리 부족, 네트워크 대역폭, 코드 비효율

**해결책**:
- **캐시 (Redis)**: 자주 보는 데이터는 메모리. DB 부담 ↓ → Day 3 [[캐시]]
- **DB 복제 (Replication)**: 읽기 전용 DB 여러 대로 분산
- **샤딩 (Sharding)**: DB를 조각으로 나눠 분산
- **인덱스**: 쿼리 빠르게 ([[색인]] 동일 원리)

→ Cloudflare·CDN 다운 = CDN 레벨 사고 / 인스타 다운 = 보통 DB 레벨 사고.

### NoSQL 깊이 — MongoDB·Redis 차이

- **MongoDB** = 디스크 저장 + 문서 형식 (JSON 같은 거). 영구 저장 OK
- **Redis** = 메모리 저장 + 키-값. 매우 빠르지만 전원 끄면 날아갈 수 있음

→ 보통 같이 씀: PostgreSQL(영구) + Redis(빠른 캐시) + MongoDB(스키마 유연 영역).

### 대기업 DB 상세 풀이 (2026-05-29 추가)

**Cassandra (카산드라)**
- 2008 Facebook → 오픈소스 Apache
- **분산 NoSQL DB**. 여러 서버에 자동 분산
- **쓰기 매우 빠름**, 복잡한 쿼리 약함
- 사용: 넷플릭스·인스타·페이스북 메시지
- 비유: **체인점 본사 시스템** (전국 지점 분산)

**RocksDB (락스DB)**
- 2012 Facebook
- **임베디드 DB** (앱 안에 박아 넣음, 별도 서버 ❌)
- 다른 큰 DB의 **내부 엔진**으로 쓰임 (Cassandra·MongoDB·Kafka 내부)
- 비유: **선풍기 모터** (DB 안의 DB)

**DynamoDB (다이나모DB)**
- 2012 AWS 출시
- **AWS 클라우드 전용 NoSQL DB**
- 자동 확장·서버리스 (사용량만큼 과금)
- 사용: Amazon·우버·에어비앤비
- 비유: **호텔 룸 서비스**

**Aurora (오로라)**
- 2014 AWS 출시
- **AWS의 RDB** (MySQL·PostgreSQL 호환)
- 일반 RDB보다 5배 빠름
- 자동 백업·복제·확장
- 비유: **호텔 풀옵션 PostgreSQL**

→ **AWS 환경에서 일하면 DynamoDB(NoSQL) + Aurora(RDB)** 조합 표준.

### RDB = 엑셀 시트 비유 (2026-05-29 추가)

**RDB (Relational Database)** = **표(테이블) 형태**로 데이터 저장.

```
users 테이블:
| id (PK) | name | email          |
|---------|------|-----------------|
| 1       | 은미  | yoon@gmail.com |
| 2       | 영희  | hee@gmail.com  |
```

- **테이블 = 엑셀 시트**
- **컬럼 = 열** (이름·나이·이메일)
- **로우 = 행** (한 명의 데이터)
- **Primary Key (PK)** = **주민등록번호** (유일 식별, NULL ❌)

**대표 RDB**: PostgreSQL·MySQL·Oracle·SQL Server·SQLite

### SQL 4개 명령어 = CRUD 매핑 (2026-05-29 추가)

```sql
SELECT * FROM users WHERE id = 123;          -- 조회 (Read / GET)
INSERT INTO users (name) VALUES ('은미');     -- 생성 (Create / POST)
UPDATE users SET name = '윤은미' WHERE id=123; -- 수정 (Update / PUT)
DELETE FROM users WHERE id = 123;            -- 삭제 (Delete / DELETE)
```

**SQL = CRUD = HTTP 메서드** 1:1:1 매핑.

### ⚠️ "PostgreSQL vs 구글 클라우드" — 비교 카테고리 다름 (2026-05-29 추가)

**가장 흔한 혼동**.

| | PostgreSQL | 구글 클라우드 (GCP) |
|---|---|---|
| **카테고리** | DB 종류 (RDB) | 클라우드 인프라 |
| **비유** | 자동차 모델 (쏘렌토) | 주차장 (차고·호텔 발레파킹) |

→ 진짜 질문은 "**PostgreSQL을 어디 둘 거냐**":

| 선택지 | 비유 | 비용 | 관리 |
|---|---|---|---|
| 내 컴퓨터 | 집 차고 | 무료 | 직접 (어려움) |
| AWS Aurora | AWS 발레파킹 | 사용량 | AWS가 |
| GCP Cloud SQL | GCP 주차장 | 사용량 | GCP가 |
| **Supabase** ⭐ | 풀옵션 호텔 주차장 | 무료 시작 | **가장 쉬움** |

**GCP 안의 DB 제품들** (참고):
- Cloud SQL = MySQL·PostgreSQL 임대
- Bigtable = NoSQL (자체)
- Spanner = 분산 RDB (자체)
- Firestore = NoSQL (Firebase 일부)

→ **은미 = Supabase 한동안 충분**. 1인 컨설팅·프로젝트윤365·Stage 4 1순위.
