# Database Basics

## 한 줄 정의

> "기록하고 조합하는 모든 데이터들의 총합" (강의 본문)

데이터를 **분류하기 좋게 구조화**하여 저장·검색·수정·삭제할 수 있게 만든 시스템.

## 핵심 아이디어

### 일상에서의 DB

- 통장의 입·출금 기록
- 도서관 도서 검색

은행앱 결제내역 조회 4단계:
1. 은행앱 열기
2. 은행 서버에 계좌정보로 접속
3. 접속 확인 / DB 검색
4. 결제 내역 확인

도서관 관리에 필요한 정보 항목:
- 책 보유 여부 / 책 정보(제목·작가·출판사) / 대여 중 여부 / 대여자 / 반납 예정일

→ **이런 항목들이 DB 컬럼이 됨**.

### CRUD — DB의 기본 4기능

```
[저장]  →  [검색]  →  [수정]  →  [삭제]
Create     Read       Update     Delete
```

→ HTTP 메서드와 1:1 매핑 ([[http-protocol]] 참조).

### 테이블 (예시)

**상품 테이블**:

| 상품번호 | 상품종류 | 가격 |
|---------|---------|------|
| 1 | 상의 | 50000 |
| 2 | 청바지 | 80000 |
| 3 | 신발 | 70000 |
| 4 | 벨트 | 10000 |

**회원 테이블**:

| 회원번호 | 이름 | 아이디 | 주소 |
|---------|------|--------|------|
| 1 | 홍길동 | A1 | 서울 |
| 2 | 철수 | B2 | 부산 |

> **핵심**: "데이터를 분류하기 좋게 저장되어 있는 형태. 사람이 분류하기 쉽게 저장하는 것이 중요" (강의 본문)

→ 서비스를 어떻게 운영할 것인가의 핵심 요소 = **DB 설계**.

### 검색 (Read)과 Query

```
[클라이언트] ── Query ──→ [DB]
```

> **Query = 데이터베이스 명령어** (강의 본문)

검색 조건 예시:
- 신상품 입고 / 핫딜 / 가격 낮은순 / 추천상품

정렬:
> "특정한 값을 기준으로 데이터를 정렬할 수 있다"

예: 신상품을 등록일자 기준 최신순 20개만, 8~10월 상품만 등.

### 데이터 타입

| 분류 | 타입 | 설명 |
|------|------|------|
| 숫자 | Int, float, double | 정수·소수 |
| 문자 | String | 문자열 (이름·주소 등) |
| 문자 | Boolean | 참/거짓 |
| 시간 | Time, Datetime | 날짜·시간 |

### 수정 (Update) 예시 — 신발 매진 케이스

매진 컬럼 추가:

| 상품번호 | 상품종류 | 가격 | 매진 |
|---------|---------|------|------|
| 1 | 상의 | 50000 | False |
| 2 | 청바지 | 80000 | False |
| **3** | **신발** | **70000** | **True** |
| 4 | 벨트 | 10000 | False |

→ Boolean 컬럼으로 상품 라이프사이클 반영.

### 물리삭제 vs 논리삭제 ⭐

같은 매진 케이스, 두 가지 접근:

**물리삭제 (Hard Delete)**: 신발 row 자체를 DB에서 제거
```
1 | 상의 | 50000 | False
2 | 청바지 | 80000 | False
~~3 | 신발 | 70000 | True~~  ← 제거
4 | 벨트 | 10000 | False
```

**논리삭제 (Soft Delete)**: 데이터를 지우지 않고 플래그로 표시
```
1 | 상의 | 50000 | False
2 | 청바지 | 80000 | False
3 | 신발 | 70000 | True   ← 매진=True 유지
4 | 벨트 | 10000 | False
```

→ **실무 핵심 패턴**:
- 물리삭제: 진짜 필요 없는 데이터 (GDPR 개인정보 삭제 등)
- 논리삭제: 통계·복구·이력 추적이 필요한 경우 (대부분의 비즈니스 데이터)

콘텐츠 후보: "매진 신발로 보는 물리삭제 vs 논리삭제 1컷"

```

```

1. **트래픽 과부하** — 다수 동시 접속으로 처리 불가

## DBMS와 DB 종류

### DBMS

> "Database Management System — 데이터베이스 운영관리를 위한 소프트웨어"

대표 DBMS:

| DBMS | 제작사 |
|------|--------|
| MySQL | Oracle |
| MariaDB | MariaDB Foundation |
| PostgreSQL | PostgreSQL Global Dev Group |
| Oracle | Oracle |

### 관계형 (RDB) vs NoSQL

- **관계형 (RDB)**: **Primary Key**라는 고유값으로 테이블 간 연결
  - 예: "B신발을 구매한 A고객" → B신발 데이터 + A고객 데이터를 PK로 연결
  - 위 대표 DBMS 4종은 모두 RDB
- **NoSQL**: 기록 단순, **대용량 처리에 유리**
  - 예: MongoDB, Redis, Cassandra (강의엔 미언급, 일반 지식)

## 왜 중요한가

- **Stage 3 어느 트랙이든 일상 도구** — 분석가는 데이터 조회·정제, 서비스개발자는 설계·운영
- **1인 컨설팅 직접 영향** — 클라이언트 데이터를 어떻게 구조화할지가 솔루션 품질의 절반
- **AI 시대에도 DB는 안 없어짐** — RAG·벡터 DB도 결국 DB 패턴 + 의미 검색

## 자주 헷갈리는 것

- **DB vs DBMS**: DB는 데이터 자체, DBMS는 관리 소프트웨어. 실무에선 거의 혼용.
- **Primary Key vs Unique Key**: 둘 다 고유값이지만 PK는 테이블당 1개·NULL 불가, Unique는 여러 개·NULL 허용.
- **RDB vs NoSQL 선택**: "데이터 간 관계가 많으면 RDB, 양이 많고 구조가 단순하면 NoSQL"이 일반론. 실무는 둘 다 쓰는 게 흔함.
- **Query = SQL?**: SQL은 RDB용 Query 언어. NoSQL은 자체 Query 문법.
