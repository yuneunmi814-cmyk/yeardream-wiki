## 🚀 예습 (5분용)

### 한 줄 요약

**프로토콜 = 통신 규약**. 7계층 위에 실제로 사는 6대 묶음 = **IP/DNS/DHCP** (네트워크 운영) + **TCP/UDP** (전송 신뢰성·속도) + **FTP/SMTP** (파일·메일) + **SSH/SSL/TLS** (보안 통신).

### 핵심 단어 5~7개

- [[TCP-UDP]] ⭐ — 신뢰성(TCP) vs 속도(UDP), 3·4-way handshake
- [[DNS]] — 도메인 → IP 번역 (인터넷 전화번호부)
- [[DHCP]] — IP 자동 할당
- [[SSH]] — 원격 접속 보안 프로토콜 (22번 포트)
- [[SSL-TLS]] — 통신 암호화 (HTTPS의 S)
- [[FTP]] — 파일 전송 (능동/수동 모드, 20·21번 포트)
- [[SMTP]] — 메일 전송

### 큰 그림

```
OSI 7계층            실제 프로토콜
─────────────────────────────────────
응용 (7층)    ←    HTTP·SMTP·FTP·SSH·DNS·DHCP
표현 (6층)    ←    JPEG·MPEG·ASCII
세션 (5층)    ←    SSL·TLS
전송 (4층)    ←    TCP·UDP ⭐ (이 강의의 메인 챕터)
네트워크(3층) ←    IP·ICMP·ARP
```

→ Day 3 4강 중 **2강**. 06강이 골격이면 이 강은 살. **TCP vs UDP가 이 강의 하이라이트**.

### 챕터 1. IP, DNS, DHCP (슬라이드 3~11)

**키워드 (강의 본문)**:
> IP (Internet Protocol) = 송수신 규약. 비신뢰성·비연결형, 주소 지정, 경로 설정(라우팅)
> DNS (Domain Name System) = 인터넷 전화번호부, 도메인 → IP 변환
> DNS 조회 단계 — 브라우저 → DNS 서버 → TLD 서버 → IP 응답
> DNS 질의 방식 — 재귀적 질의 / 반복적 질의
> DHCP (Dynamic Host Configuration Protocol) = IP·게이트웨이·네임서버 자동 할당
> DHCP 할당 종류 — 수동·자동·동적

✦ **강의자 풀이 추정** (LLM 보강)

**IP·DNS·DHCP = 인터넷이 동작하기 위한 3대 기반 프로토콜**.

**[[IP-주소|IP]]** — 06강에서 다룬 그 IP. 프로토콜 측면에서 보면:
- **비신뢰성**: 패킷이 도착한다는 보장 없음 (TCP가 4층에서 보완)
- **비연결형**: 매번 새로 연결하는 게 아니라 그냥 던짐
- 역할: **주소 지정** + **경로 설정(라우팅)**

**[[DNS]] — 도메인 → IP 번역** ⭐:

`www.google.com` 같은 도메인을 `142.250.207.78` 같은 IP로 바꿔주는 시스템. **인터넷 전화번호부**.

DNS 조회 4단계:
```
1. 브라우저: "www.google.com IP 좀"
   ↓
2. Local DNS 서버: TLD(.com) 서버에 요청
   ↓
3. TLD 서버: "Google authoritative DNS에 물어봐"
   ↓
4. Authoritative DNS: "142.250.207.78"
```

DNS 질의 2종:
- **재귀적 질의**: Local DNS가 다 처리. 클라이언트는 답만 받음 (대부분 이 방식)
- **반복적 질의**: 클라이언트가 매번 다음 서버에 직접 물어봄

**[[DHCP]] — IP 자동 할당**:

집에 와이파이 처음 연결하면 자동으로 IP 받는데, 그게 DHCP. 사용자는 IP 입력 안 함.

DHCP 할당 종류:
- **수동**: 관리자가 고정 IP (서버·프린터 등)
- **자동**: 한 번 받으면 영구 (고정 임대)
- **동적**: 일정 시간(lease period) 후 회수·재할당. **가장 흔함**

**🎯 이 부분의 핵심 메시지**: 인터넷 시작 = **DHCP로 IP 받고 + DNS로 도메인 IP 알아내기**. 사용자는 인지 못 하지만 와이파이 켜는 순간 둘 다 일어남.

⚠️ **흔한 오해**: "DNS 변경하면 인터넷 빨라진다". 일부 맞음 (`1.1.1.1`, `8.8.8.8` 같은 공용 DNS가 ISP DNS보다 빠르거나 안전할 수 있음). 단 광케이블 자체가 느리면 효과 없음.

### 챕터 2. TCP / UDP ⭐ (슬라이드 12~20)

**키워드 (강의 본문)**:
> TCP (Transmission Control Protocol) = 신뢰성 있는 데이터 전송
> 3-way handshake — 연결 성립 (SYN·SYN-ACK·ACK)
> 4-way handshake — 연결 해제
> TCP 특징 — 연결형, 흐름·혼잡 제어, 높은 신뢰성, Full-Duplex, 느림
> UDP (User Datagram Protocol) = 빠른 데이터 전송
> UDP 특징 — 비연결형, 신뢰성 낮음, TCP보다 빠름

✦ **강의자 풀이 추정** (LLM 보강) — **Day 3 핵심** ⭐

**TCP와 UDP는 [[OSI-7계층]] 4층(전송 계층) 두 프로토콜**. 인터넷의 거의 모든 통신이 둘 중 하나.

**TCP** = "**신뢰성 우선**":

```
   클라이언트                  서버
       │  ─── SYN ─────────→  │      "연결할래?"
       │  ←── SYN+ACK ────── │      "응 가능, 너도 OK?"
       │  ─── ACK ─────────→  │      "OK 시작"
       ↓                       ↓
       === 3-way handshake 완료, 데이터 송수신 ===
       ↓                       ↓
       │  ─── FIN ─────────→  │      "끊자"
       │  ←── ACK ────────── │      "OK 곧 끊을게"
       │  ←── FIN ────────── │      "이제 끊자"
       │  ─── ACK ─────────→  │      "OK"
       ↓                       ↓
       === 4-way handshake 완료, 연결 해제 ===
```

특징:
- **연결형**: 3-way handshake로 연결 확립
- **신뢰성 보장**: 패킷 분실 시 재전송, 순서 보장
- **흐름·혼잡 제어**: 받는 쪽이 느리면 천천히
- **Full-Duplex**: 양방향 동시 송수신
- **속도**: UDP보다 느림

**UDP** = "**속도 우선**":

악수 없이 그냥 던짐. 도착 보장 ❌, 순서 보장 ❌, 빠름 ✅.

| | TCP | UDP |
|---|---|---|
| **연결** | 연결형 (3-way) | 비연결형 |
| **순서** | 보장 | 보장 ❌ |
| **수신 확인** | 함 | 안 함 |
| **통신** | 1:1 | 1:1, 1:N, N:N |
| **신뢰성** | 높음 | 낮음 |
| **속도** | 느림 | 빠름 |
| **실사용** | 웹(HTTP)·메일·SSH·파일전송 | 음성통화·게임·영상스트리밍·DNS |

**🎯 이 부분의 핵심 메시지**: 인터넷 통신은 **신뢰성(TCP)이냐 속도(UDP)냐** 둘 중 하나. 웹·메일은 TCP, 실시간 영상·게임은 UDP.

⚠️ **흔한 오해**: "UDP는 쓸모없는 거 아닌가". 영상 통화·게임은 패킷 한두 개 잃는 게 1초 지연보다 나음 → UDP가 합리적. DNS도 UDP (응답 짧고 빠름).

⚠️ **흔한 오해 2**: "3-way handshake가 왜 3번?". 2번이면 송신자만 확인, 4번이면 낭비. 3번이 양쪽 다 확인하는 **최소 횟수**.

### 챕터 3. FTP (슬라이드 21~30)

**키워드 (강의 본문)**:
> FTP (File Transfer Protocol) = 파일 전송 프로토콜
> 21번 포트 = 신호/명령 제어, 20번 포트 = 데이터 전송
> 3-way handshake 후 시작
> FTP 종류 — FTP / TFTP / SFTP
> FTP 모드 — 능동(Active) / 수동(Passive)
> SFTP (SSH File Transfer Protocol) — SSH 데이터 스트림 통해 파일 전송

✦ **강의자 풀이 추정** (LLM 보강)

**[[FTP]]** = 컴퓨터 간 **파일 송수신 전용 프로토콜**. 1971년 등장, 인터넷 초기부터 존재.

**포트 2개 사용**:
- **21번 포트**: 명령 채널 ("이 파일 줘", "여기로 보내")
- **20번 포트**: 데이터 채널 (실제 파일 송수신)

**FTP 모드 2종**:
- **능동 모드 (Active)**: 서버가 클라이언트에 능동적으로 데이터 연결. 방화벽 문제 발생 가능
- **수동 모드 (Passive)**: 클라이언트가 두 채널 모두 설정. 방화벽 친화적. **현재 표준**

**FTP 종류 3가지**:

| | FTP | TFTP | SFTP |
|---|---|---|---|
| **전송** | TCP | UDP | TCP (SSH 안에) |
| **인증** | ID/PW | 없음 | SSH 키 |
| **암호화** | ❌ (평문) | ❌ | ✅ |
| **포트** | 20·21 | 69 | 22 (SSH 포트) |
| **용도** | 일반 파일 | 부팅 이미지·라우터 설정 | 보안 파일 전송 |

⚠️ **흔한 오해**: "SFTP = FTPS". 다름.
- **SFTP** = **SSH** 위에 올린 파일 전송 (다른 프로토콜)
- **FTPS** = 기존 FTP에 **SSL/TLS** 덧붙인 것

**🎯 이 부분의 핵심 메시지**: FTP는 **2-포트 구조** (명령 21번 + 데이터 20번). 평문 전송이 약점 → 보안 필요하면 SFTP.

### 챕터 4. SMTP / POP3 / IMAP (슬라이드 31~36)

**키워드 (강의 본문)**:
> SMTP (Simple Mail Transfer Protocol) = 전자우편 전송
> SMTP 메일 처리 — MUA → MSA → MTA → MX → MDA
> POP3 = 메일 서버에서 로컬 PC로 이메일 수신 (다운로드)
> IMAP (Internet Messaging Access Protocol) = 비연결 원격 편지함 접근
> SMTP vs IMAP vs POP3 — 보내기 vs 서버 관리 vs 다운로드

✦ **강의자 풀이 추정** (LLM 보강)

**이메일 3대 프로토콜** — 역할이 명확히 분리:

| | 역할 | 비유 |
|---|---|---|
| **SMTP** | 메일 **보내기** | 우체통에 편지 넣기 |
| **POP3** | 메일 **서버에서 내려받기** (다운로드 후 서버에서 삭제) | 우체국 사서함에서 편지 꺼내가기 |
| **IMAP** | 메일 **서버에서 보기·관리** (서버에 남김) | 사서함에서 편지 보고 다시 두기 |

**IMAP vs POP3 — 현대 표준은 IMAP**:
- POP3: 한 기기에서만 메일 봄. 다른 기기 동기화 ❌
- IMAP: 모든 기기 동기화. 폰·노트북·태블릿에서 같은 메일함 봄. **Gmail·iCloud 다 IMAP**

**SMTP 메일 처리 경로** (전문 용어):
```
MUA (Mail User Agent: 클라이언트 앱, 예: Mail.app)
   ↓ SMTP
MSA (Mail Submission Agent)
   ↓
MTA (Mail Transfer Agent) → MX 레코드 조회 → 다음 MTA
   ↓
MDA (Mail Delivery Agent) → 수신자 mailbox
```

**🎯 이 부분의 핵심 메시지**: 메일은 **보내기(SMTP) + 받기(POP3/IMAP)** 따로. 현재는 IMAP이 표준 (다기기 동기화 시대).

### 챕터 5. TELNET / SSH (슬라이드 37~41)

**키워드 (강의 본문)**:
> TELNET = 원격 접속 인터넷 표준 프로토콜
> SSH (Secure Shell) = 원격 호스트 보안 접속 프로토콜
> SSH 특징 — 데이터 암호화, 트래픽 압축, 22번 포트
> Public Key / Private Key 사용
> SSH 동작 — 양쪽 키 보유, 인증 후 안전한 데이터 교환

✦ **강의자 풀이 추정** (LLM 보강)

**TELNET vs SSH — 같은 목적, 다른 시대**:

| | TELNET | SSH |
|---|---|---|
| **등장** | 1969 | 1995 |
| **암호화** | ❌ (평문) | ✅ (강력) |
| **포트** | 23 | 22 |
| **사용** | 거의 사장 (사내 장비 일부) | **사실상 표준** |

**[[SSH]]** ⭐ = 원격 접속의 사실상 표준. Day 2 [[1주차-2일차-07-프로세스와-네트워크]]에서도 등장.

**SSH 핵심 = Public/Private Key 쌍**:
- **Public Key (공개키)**: 누구나 봐도 됨. 서버에 등록
- **Private Key (개인키)**: 본인 컴퓨터에만. **절대 노출 금지** ⚠️

**SSH 인증 흐름**:
```
1. 클라이언트가 서버에 접속 요청
2. 서버: "내 public key 줄게"
3. 클라이언트: 자신의 private key로 서명한 데이터 보냄
4. 서버: 등록된 클라이언트 public key로 검증
5. 검증 성공 → 암호화된 채널 수립
```

**실무 사용 예** (Day 2 SSH 챕터와 연결):
- GitHub 푸시할 때 `git push` → SSH 키 사용
- 원격 서버 접속 `ssh user@server.com`
- 클라우드 EC2·서버 관리

**🎯 이 부분의 핵심 메시지**: SSH는 **22번 포트 + Public/Private Key 쌍**으로 안전한 원격 접속. Private Key 유출되면 즉시 모든 게 뚫림.

⚠️ **흔한 오해**: "TELNET 아직 쓸 만". 평문이라 비밀번호 그대로 보임. 외부 노출 금지.

### 챕터 6. SSL / TLS / HTTPS (슬라이드 42~47)

**키워드 (강의 본문)**:
> SSL (Secure Sockets Layer) = 암호화 기반 인터넷 보안 프로토콜
> TLS (Transport Layer Security) = SSL보다 더 안전한 버전
> SSL vs TLS — 같은 목적, TLS가 후속 버전
> HTTPS (HTTP Secure) = HTTP + SSL/TLS, 데이터 유출 방지

✦ **강의자 풀이 추정** (LLM 보강)

**SSL과 TLS는 사실상 같은 것 (SSL이 옛 이름, TLS가 현재 이름)**:
- SSL 1.0~3.0 → TLS 1.0 → 1.1 → 1.2 → 1.3 (현재 표준)
- 일반인은 "SSL/TLS" 통칭, 정확히는 **모두 TLS**

**역할** = OSI 5층(세션) 또는 6층에서 **암호화 채널 제공**.

**[[SSL-TLS]] 핸드셰이크 (간략)**:
```
1. 클라이언트: "안녕, 내가 지원하는 암호 알고리즘은 이거야"
2. 서버: "이걸로 가자 + 내 인증서(public key 포함)"
3. 클라이언트: 인증서 검증 → 대칭키 만들어서 서버 public key로 암호화 → 전송
4. 서버: 자기 private key로 복호화 → 대칭키 획득
5. 이제 양쪽 다 대칭키 보유 → 빠른 암호화 통신 시작
```

**[[HTTPS]] = HTTP + TLS** ⭐:
- HTTP는 평문. 비밀번호·카드번호 그대로 보임 → 큰일
- HTTP를 TLS 채널 위에 올리면 HTTPS. 중간에서 못 봄
- 브라우저 주소창 자물쇠 = HTTPS

**🎯 이 부분의 핵심 메시지**: SSL/TLS는 **통신 암호화 프로토콜**. HTTPS = HTTP를 TLS로 감싼 것. 현대 웹의 사실상 강제 표준.

⚠️ **흔한 오해**: "HTTPS면 다 안전". TLS는 **전송 구간**만 암호화. 서버에 도착한 데이터는 서버가 해킹되면 그대로 노출. + Phishing 사이트도 HTTPS 가능 (자물쇠 = 신뢰가 아님).

## 💡 일상 비유

### 비유 1: TCP vs UDP = 등기 vs 일반 우편

- **TCP (등기)**: "받았어요" 확인 받고, 분실되면 재발송, 순서대로 도착. **느림·확실함**. → 계약서·웹사이트·메일
- **UDP (일반 우편)**: 그냥 던짐. 도착했는지 모르고, 순서도 안 맞을 수 있음. **빠름·불확실**. → 광고지·실시간 영상·게임

### 비유 2: DNS = 카톡 친구 검색

`www.google.com` → IP 주소 = **친구 이름** → **전화번호**.

카톡에서 "은미" 검색하면 카톡이 내부적으로 "은미의 user ID는 12345"라고 변환. DNS가 도메인 ↔ IP에 하는 일과 같음.

### 비유 3: SSH 키 = 우리 집 열쇠

- **Public Key (현관 자물쇠)**: 누구나 봐도 됨. 집 앞에 박혀 있음
- **Private Key (열쇠)**: 본인만 가짐. **잃어버리면 큰일** ⚠️
- 자물쇠 + 열쇠 한 쌍만 맞아야 문이 열림 = SSH 인증
