# 01 / 앱·웹 서비스를 위한 모든 것 — 교재 버전

## 🚀 예습 (5분용)

### 한 줄 요약

**HTTP 약속 위에서 클라이언트 ↔ Web Server ↔ Back-end ↔ DB가 대화**하며 디지털 명함 같은 서비스를 만든다. CRUD = GET·POST·PUT·DELETE.

### 핵심 단어 5~7개

- [[앱-웹-서비스]] — 통합 노트
- [[HTTP-프로토콜]] — HTTP 분해·메서드
- [[REST-API]] — REST API + URI/URL
- [[동기-vs-비동기]] — 동기/비동기
- [[파싱]] — JSON·XML 파싱
- **WAS (Web Application Server)** — 웹서버 + 백엔드
- **Endpoint** — 자원을 가리키는 URL

### 큰 그림

```
[클라이언트(폰·브라우저)] ⇄ HTTP ⇄ [Web Server] ⇄ [Back-end] ⇄ [DB]
       프론트엔드                                            ↑
       UI/UX                                          다음 강 (3강)
       동기/비동기
```

→ Day 1 시리즈 **2강**. 1강의 "클라이언트-서버"를 본격 펼침. 디지털 명함 앱이 시리즈 전체 러닝 예제.

**키워드 (강의 본문)**:

명함촬영 → 명함 사진 정보전송 → DB 직접입력 → 디지털 명함

✦ **강의자 풀이 추정**
이 시리즈의 모든 개념을 **디지털 명함 앱**으로 끌어옴. 리멤버 앱 비슷한 OCR + 자동 정리. 사용자 NyangTalk 만든 경험과 흐름 비슷.

**🎯 핵심 메시지**: 6강 시리즈 러닝 예제 = 디지털 명함 앱. 머릿속에 깔고 가기.

### 챕터 2. HTTP 통신 (슬라이드 2~6)

**키워드 (강의 본문)**:
- **통신**: 컴퓨터 간 Communication. 1요청·1응답 원칙
- **HTTP** 분해:
  - **H**yper — 기기 무관 문자 통신
  - **T**ext — OS 무관 문자 통신
  - **T**ransfer — 브라우저 무관 통신
  - **P**rotocol — 약속된 룰
- **Hyper text 종류**: **JSON**(현대 표준), **XML**(레거시)
- **파싱**: 받은 문자를 잘라 의미 단위로 사용 → [[파싱]]

✦ **강의자 풀이 추정**
HTTP = 1990년대 초 팀 버너스리가 만든 통신 약속. 기기·OS·브라우저 무관.

JSON 예시 (명함):
```json
{ "id": 1, "name": "홍길동", "company": "엘리스", "phone": "010-1234-5678" }
```

→ 사용자가 Claude API·Meta 광고 API 호출할 때 응답이 다 JSON. 일상.

**🎯 핵심 메시지**: HTTP = 인터넷 공통어. JSON이 현대 데이터 형식 표준.

⚠️ **흔한 오해 1**: "HTTP = 웹사이트만 쓰는 것" → 모바일 앱·AI Agent·IoT 다 HTTP.

⚠️ **흔한 오해 2**: "HTTPS는 HTTP의 새 버전" → HTTPS = HTTP + 암호화(S=Secure). 같은 통신.

📎 자세히: [[HTTP-프로토콜]], [[파싱]]

### 챕터 3. WAS 구조 (슬라이드 7~8)

**키워드 (강의 본문)**:
```
[클라이언트] → 요청 → [Web Server] ⇄ [Back-end] ⇄ [DataBase]
            ← 응답 ←   (http 통신)      동작 코드      각종 정보 저장
```
- **Endpoint**: 웹 서버 주소. `www.google.com`·`www.naver.com` 같은 URL

✦ **강의자 풀이 추정**
거의 모든 웹·앱의 **4단 표준 구조**. Express(Node.js)·Django(Python)는 Web Server + Back-end를 합쳐 처리.

명함 저장 흐름:
1. 클라이언트 → `POST /cards` JSON 전송
2. Web Server가 받음
3. 백엔드가 파싱 + DB에 INSERT
4. 응답 → 클라이언트 화면 "저장 완료"

**🎯 핵심 메시지**: 4단 구조 (클라이언트→웹서버→백엔드→DB)가 거의 모든 서비스의 골격.

⚠️ **흔한 오해**: "Web Server = Back-end" → 분리 개념. 실무에선 같이 처리되어 경계 흐릿.

📎 자세히: [[앱-웹-서비스]], [[데이터베이스-기초]]

### 챕터 4. Front-end (UI/UX·Grid) (슬라이드 9~14)

**키워드 (강의 본문)**:
- **Front-end 4책임**: UI/UX, 독립 기능, 서버 통신, **Analytics**
- **UI** (User Interface) = 사용자가 만나는 화면. 디자인 요소
- **GUI 메모**: 현대엔 GUI/UI 구분 없이 통용
- **UX** (User Experience) = 서비스 성격에 맞는 편리 경험. 짧은 경로 vs 긴 경로
- **반응형** = 다양한 사이즈 + 동일 정보 제공
- **Grid System** = 12 Columns 분할. 모바일은 Component System

✦ **강의자 풀이 추정**
프론트엔드 = 사용자가 만지는 모든 것. **Analytics가 흔히 간과되는 4번째 책임** (Google Analytics·Mixpanel 같은 도구로 사용자 행동 추적).

UX 예시 — 카메라 도달 경로:
- 긴 경로: 로그인 → 메인 → 메뉴 → 기능 → 촬영 (4단계)
- 짧은 경로: 로그인 → 메인 하단 버튼 → 촬영 (2단계)

**Grid System**: 12 Columns가 웹 표준 (Bootstrap·Tailwind). 인스타 프로필 3×3 그리드가 모바일 사례.

→ 사용자 SwiftUI 경험 = 프론트엔드 그 자체. 이 강의는 "익숙한 거 멘탈 모델 정리".

**🎯 핵심 메시지**: 프론트엔드 = UI/UX + 독립 기능 + 통신 + Analytics 4책임.

⚠️ **흔한 오해**: "짧은 경로가 무조건 좋은 UX" → 거짓. 계정 탈퇴는 일부러 어렵게 만듦 (실수 방지).

### 챕터 5. HTTP 메서드 4종 (슬라이드 15~19)

**키워드 (강의 본문)**:

| 메서드 | 기능 |
|--------|------|
| **GET** | 기존 정보·페이지 읽기 |
| **POST** | 정보 저장·전달 |
| **PUT, PATCH** | 기존 정보 수정 |
| **DELETE** | 정보 삭제 |

✦ **강의자 풀이 추정**
HTTP 메서드 = **CRUD와 1:1**:

| HTTP | CRUD | 명함 예시 |
|------|------|-----------|
| POST | Create | 명함 저장 |
| GET | Read | 명함 조회 |
| PUT/PATCH | Update | 명함 수정 |
| DELETE | Delete | 명함 삭제 |

**🎯 핵심 메시지**: HTTP 메서드 4종 = CRUD 1:1. 메서드로 의도 표현 = RESTful의 핵심.

⚠️ **흔한 오해 — 강의 본문 부정확** (PUT vs PATCH):

강의 본문: "PUT은 보안 이슈로 PATCH가 등장"
**사실은 PUT = 전체 교체, PATCH = 부분 수정**. RFC 5789(2010) 표준 추가. 보안과 무관.

→ [[HTTP-프로토콜]] PUT vs PATCH 보강 섹션 참조.

📎 자세히: [[HTTP-프로토콜]]

### 챕터 6. 동기 vs 비동기 (슬라이드 20~22)

**키워드 (강의 본문)**:
- **동기**: 통신 결과와 함께 움직임 (페이지 이동·정보 표시)
- **비동기**: 통신 결과와 상관없이 움직임 (좋아요·댓글)
- **영업사원 비유** ⭐: 명함 디지털화 안 되어도 영업 활동은 독립적으로 진행

✦ **강의자 풀이 추정**
- **동기** = 응답 기다림 → 화면이 응답 후 그려짐 (페이지 로딩)
- **비동기** = 안 기다림 → 응답 오면 그때 처리 (좋아요 즉시 빨갛게)

→ AI Agent·자동화 워크플로우 설계의 핵심 결정 포인트.

**🎯 핵심 메시지**: 동기 = 기다림, 비동기 = 안 기다림. UX 부드러움의 비결.

⚠️ **흔한 오해 1**: "비동기 = 빠르다" → 작업 시간 동일. "기다리지 않음"이 본질.

⚠️ **흔한 오해 2**: "비동기 = 멀티 스레드" → 다름. JavaScript는 단일 스레드에서도 비동기 가능.

📎 자세히: [[동기-vs-비동기]]

---

### 챕터 7. 백엔드 + REST API (슬라이드 23~27)

**키워드 (강의 본문)**:
- **백엔드** = 서버 측 동작 코드. **비즈니스 로직** 처리 (데이터 생성·표시·저장·변경)
- 프론트가 GET 요청 → 백엔드도 **반드시 GET**으로 호응
- **REST API**: 약속된 주소(URI) + 각 주소에 기능(함수). 예: `https://school.elice.io/courses/26028/info`
- **URI vs URL**: URI가 URL보다 더 큰 의미

✦ **강의자 풀이 추정**
백엔드 일 = "보이지 않는 로직 처리". 회원가입 1초 안에 8단계 처리:
이메일 형식 → 중복 확인 → PW 해싱 → 인증 메일 → DB 저장 → 토큰 발급 → 응답.

**RESTful 설계** = 같은 URL 패턴, 메서드로 동작 분기:
```
GET    /cards       → 모든 명함
POST   /cards       → 새 명함
GET    /cards/42    → 42번 명함
PUT    /cards/42    → 42번 교체
DELETE /cards/42    → 42번 삭제
```

**🎯 핵심 메시지**: REST API = URI + HTTP 메서드 조합. 현대 웹 API 표준.

⚠️ **흔한 오해 — 강의 본문 보완** (URI vs URL):

강의 본문: "URI가 URL보다 큰 의미"
**정확히는 URI = URL ∪ URN**. URL은 위치 기반, URN은 이름 기반(예: ISBN). 실무는 거의 동의어.

📎 자세히: [[REST-API]]

## 💡 일상 비유

### 비유 1 — 카카오톡 메시지 (HTTP)

내 폰(클라이언트) → 카카오 서버 → 친구 폰(클라이언트). 둘이 직접 연결 X, 서버 거침. 친구 폰 꺼져 있어도 메시지 안 사라짐 (서버 저장). HTTP의 정확한 동작.

### 비유 2 — 인스타 좋아요 (비동기)

❤️ 누르면 즉시 빨갛게 변함 = 비동기. 서버 응답을 안 기다리고 UI 먼저 반영. 만약 동기였다면 0.5초 멍하다 빨개짐 = 답답.

## 이해 체크 (은미, 2026-05-29)

- [x] **앱 / 웹 / 서비스 구분** — 앱=설치, 웹=브라우저, 서비스=둘 다 + 서버까지 (인스타 = 앱 + minecraft.net 같은 웹 + 백엔드 모두)
- [x] **HTTP = 클라이언트·서버 대화 약속** — "GET 인벤토리 보여줘 / POST 다이아 넣어줘"
- [x] **HTTP 메서드 4개 = CRUD 매핑** (GET=R / POST=C / PUT=U / DELETE=D)
- [x] **HTTP 헤더** = 부가 정보 라벨 (보낸 사람·시간·인증·데이터 형식)
- [x] **HTTP 상태코드** = 결과 알리는 3자리 (200 성공·400 클라잘못·500 서버잘못)
- [x] **REST API = HTTP 4메서드 + URL로 만든 깔끔한 통로** (네 표현: 자원을 다른 사람이 가져다 쓸 수 있게)
- [x] **URI ≈ URL** (실무에선 같이 씀, URI가 좀 더 큰 개념)
- [x] **동기 = 끝까지 기다림 / 비동기 = 시켜놓고 다른 거** (인스타 DM 보내고 기다리기 vs 영상 4개 동시 업로드)
- [x] **JSON vs XML** — 둘 다 데이터 형식. JSON 가벼움 (현재 90%), XML 무거움 (옛 시스템)
- [x] **JSON·XML = 정보 포장지 / 프로그래밍 언어 = 포장지 만들고 풀고 화면 표시** (재료 vs 레시피)
- [x] **WAS = 사용자마다 다른 페이지 그때그때 만들어 응답** (인스타 피드 = 사람마다 다름)
- [x] **웹 서버(Nginx·Apache) vs WAS(Node.js·Django·FastAPI·Spring·Rails)** — 정적 vs 동적, 기술 스택 다름
- [x] **AI 시대 표준 스택 = Python + FastAPI + PostgreSQL + React** ⭐ (트랙 결정 시 참고)
- [x] **Endpoint = API의 URL 1개 = 서버 한 기능 출입구** (github.com/유저ID = Endpoint, 인스타 "프로필 보기"·"DM 보내기" 게이트)
