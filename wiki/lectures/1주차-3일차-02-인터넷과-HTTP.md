# 02 / 인터넷과 HTTP — 교재 버전

> **4섹션 구조** → [[_meta/학습-워크플로우]]
> Day 1 [[HTTP-프로토콜]]의 본격 deep dive. 상태코드 전부 + CORS 신규.

---

## 🚀 예습 (5분용)

### 한 줄 요약

**HTTP = 웹의 표준 통신 프로토콜**. OSI 7계층의 응용 계층에 살고, TCP/IP 4계층 모델 위에서 동작. Request·Response 구조 + 5종 상태코드(1xx~5xx) + 보안 정책(CORS/SOP)이 핵심.

### 핵심 단어 5~7개

- [[TCP-IP-4계층]] ⭐ — OSI 7계층의 실용 버전
- [[HTTP-프로토콜]] — 누적 (Day 1 기존)
- [[HTTP-상태코드]] — 1xx 정보·2xx 성공·3xx 리다이렉트·4xx 클라 에러·5xx 서버 에러
- [[CORS]] / [[SOP]] — Same-Origin Policy, 웹 보안 핵심
- **HTTP 메서드**: GET·POST·PUT·DELETE → CRUD 매핑
- **Request·Response** — 클라이언트-서버 모델
- **인트라넷·엑스트라넷·인터넷** — 접근성 분류

### 큰 그림

```
OSI 7계층  ↔  TCP/IP 4계층

응용·표현·세션 → 응용 계층 (HTTP·DNS·FTP·SSH)
전송             → 전송 계층 (TCP·UDP)
네트워크         → 인터넷 계층 (IP)
데이터링크·물리  → 네트워크 액세스 계층 (Ethernet·Wi-Fi)

위에 HTTP가 살고 → Request·Response 주고받고 → 상태코드로 결과 알림
                                      ↓
                  CORS·SOP가 다른 출처 접근 막거나 허용
```

→ Day 3 4강 중 **3강**. 1·2강 기반 위에 HTTP를 심화. **이 강의가 1주차 가장 누적 큰 페이지**.

### 챕터 1. TCP/IP (슬라이드 3~14)

**키워드 (강의 본문)**:
> TCP/IP = 디바이스 연결 프로토콜 집합, 데이터가 정확한 목적지에 도달
> TCP — 신뢰성·다중화·연결지향
> OSI 7계층 vs TCP/IP 4계층 — 4계층은 7계층의 실용 버전
> 4계층: 응용·전송·인터넷·네트워크 액세스
> 응용 계층 — Data/Message
> 전송 계층 — Segment, 통신 노드 간 연결 제어
> 인터넷 계층 — Packet, 최종 목적지 연결성 제공
> 네트워크 액세스 계층 — Frame, 물리적 전송
> 인터넷 — 시간·거리 제약 없는 전 세계 통신
> 인트라넷·엑스트라넷 — 사설망·외부 협력사

✦ **강의자 풀이 추정** (LLM 보강)

**TCP/IP 4계층 = OSI 7계층의 실용 버전** ⭐

OSI 7계층은 학술적·이론적 모델. 실제 인터넷은 **TCP/IP 4계층**으로 동작.

```
OSI 7계층                       TCP/IP 4계층
─────────────────              ──────────────────────
7 응용       ┐
6 표현       ├──────→  4 응용 계층 (Application)        HTTP·DNS·SMTP·SSH
5 세션       ┘                                          데이터 단위: Data/Message
─────────────────              ──────────────────────
4 전송       ───────→  3 전송 계층 (Transport)          TCP·UDP
                                                       데이터 단위: Segment
─────────────────              ──────────────────────
3 네트워크   ───────→  2 인터넷 계층 (Internet)         IP·ICMP·ARP
                                                       데이터 단위: Packet
─────────────────              ──────────────────────
2 데이터링크 ┐
1 물리       ┴──────→  1 네트워크 액세스 (Network       Ethernet·Wi-Fi
                       Access Layer)                   데이터 단위: Frame
─────────────────              ──────────────────────
```

**TCP/IP 4계층 각각**:

1. **응용 계층** (OSI 5·6·7층 통합)
   - 사용자가 직접 만나는 서비스
   - HTTP(웹) · DNS(도메인) · SMTP(메일) · SSH(원격) · FTP(파일)
   - 데이터 단위: **Data/Message**

2. **전송 계층** (OSI 4층)
   - 종단 간 통신 제어
   - TCP(신뢰) · UDP(속도)
   - 데이터 단위: **Segment** (TCP) / Datagram (UDP)

3. **인터넷 계층** (OSI 3층)
   - 목적지까지 경로 (IP 주소)
   - 데이터 단위: **Packet**

4. **네트워크 액세스 계층** (OSI 1·2층 통합)
   - 물리적 전송, MAC 주소
   - 데이터 단위: **Frame**

**인터넷 / 인트라넷 / 엑스트라넷**:

| | 인터넷 | 인트라넷 | 엑스트라넷 |
|---|---|---|---|
| **접근성** | 공개, 제한 ❌ | 비공개, 제한적 | 한정적 공개 |
| **사용자** | 전 세계 누구나 | 회사 내부 직원 | 협력사·고객 |
| **목적** | 일반 정보 교환 | 보안 + 내부 협업 | 비즈니스 제휴 |
| **예** | google.com | 사내 인트라넷 페이지 | B2B 협업 포털 |

**🎯 이 부분의 핵심 메시지**: 인터넷의 실제 동작 모델 = **TCP/IP 4계층**. OSI 7층은 개념 이해용, 4층은 실무.

⚠️ **흔한 오해**: "OSI 7계층과 TCP/IP 4계층 중 어느 게 맞나?". 둘 다 맞음. **목적이 다름** — OSI는 이론·교육, TCP/IP는 실무.

### 챕터 2. 프로토콜 (슬라이드 15~21)

**키워드 (강의 본문)**:
> 프로토콜 = 컴퓨터 통신 규약. 신호 처리·오류·암호·인증·주소
> 네트워크 아키텍처 = 프로토콜의 집합 (복잡한 시스템 → 단순화)
> 프로토콜 3요소: 구문(Syntax)·의미(Semantics)·시간(Timing)
> 프로토콜 기능: 단편화·재합성·캡슐화·연결/흐름/오류 제어·순서 결정·동기화·다중화
> 캡슐화 (Encapsulation) — 통신 프로토콜 정보를 데이터에 추가
> 역캡슐화 (Decapsulation) — 데이터에서 제거
> 각 계층별 프로토콜 (응용=HTTP/SMTP, 전송=TCP/UDP, 네트워크=IP 등)

✦ **강의자 풀이 추정** (LLM 보강)

**프로토콜 = 통신 규약**. 사람으로 치면 "한국말로 한다·경어 쓴다·인사부터 한다" 같은 약속.

**프로토콜 3요소**:
- **구문 (Syntax)**: 데이터 형식·신호 레벨 ("Header 12바이트 + Body" 같은 약속)
- **의미 (Semantics)**: 각 비트가 뭘 뜻하는가
- **시간 (Timing)**: 통신 속도·순서

**캡슐화 (Encapsulation)** ⭐ — 계층 통과 시 헤더 붙이기:

```
사용자: "Hello"

응용 계층  → [HTTP Header][Hello]                    ← HTTP가 헤더 붙임
전송 계층  → [TCP Header][HTTP Header][Hello]        ← TCP가 또 붙임
인터넷     → [IP Header][TCP Header][HTTP H][Hello]  ← IP가 또
네트워크   → [Frame][IP H][TCP H][HTTP H][Hello][CRC]← Frame이 또
물리       → 01010101...                              ← 비트로 변환
```

**역캡슐화 (Decapsulation)** — 수신 측은 역순으로 헤더 떼면서 위 계층으로 전달. 양파 까듯이.

**🎯 이 부분의 핵심 메시지**: 프로토콜은 **계층마다 자기 헤더를 붙이고 떼는 캡슐화 구조**. 각 계층은 자기 일만 함.

### 챕터 3. HTTP (슬라이드 22~28)

**키워드 (강의 본문)**:
> HTTP = Hyper Text Transfer Protocol, 클라이언트-서버 모델
> Stateless 프로토콜 — 서버가 상태·데이터를 유지하지 않음
> HTTP 구조: Start Line + Header + Empty Line + Message Body
> HTTP 헤더 — 키-값 쌍, 요청·응답 부가 정보
> HTTP 메서드: GET(조회)·POST(등록)·PUT(수정)·DELETE(삭제)
> CRUD = Create·Read·Update·Delete, 데이터 처리 4기능
> Request & Response — 서버-클라이언트 데이터 교환 방법

✦ **강의자 풀이 추정** (LLM 보강) — Day 1 [[HTTP-프로토콜]] 누적 ⭐

**HTTP 구조 (Request 기준)**:

```
GET /api/users HTTP/1.1                ← Start Line (메서드 + URI + 버전)
Host: example.com                       ← Header
User-Agent: Mozilla/5.0
Accept: application/json
                                        ← Empty Line (빈 줄로 헤더-본문 구분)
{"page": 1, "limit": 20}                ← Message Body (POST·PUT일 때 데이터)
```

**HTTP 메서드 vs CRUD** (Day 1 누적):

| HTTP 메서드 | CRUD | DB 동작 | 예시 |
|---|---|---|---|
| **GET** | Read | 조회 | `/users/123` 사용자 정보 조회 |
| **POST** | Create | 생성 | `/users` 새 사용자 등록 |
| **PUT** | Update | 수정 | `/users/123` 사용자 전체 수정 |
| **PATCH** | Update (부분) | 부분 수정 | `/users/123` 이름만 수정 |
| **DELETE** | Delete | 삭제 | `/users/123` 사용자 삭제 |

**Stateless** ⭐ — HTTP의 중요한 특성:
- 서버가 **이전 요청을 기억하지 않음**. 모든 요청은 독립적.
- 클라이언트가 매번 자기 정체를 알려줘야 함 (쿠키·세션·토큰으로 보완)
- 09강 보안 챕터에서 이 약점을 보완하는 4가지 메커니즘 학습

**🎯 이 부분의 핵심 메시지**: HTTP는 **Stateless · Request/Response** 모델. 메서드(GET·POST·PUT·DELETE) = CRUD 매핑.

⚠️ **흔한 오해**: "HTTP는 웹사이트만 쓰는 거". 모든 REST API가 HTTP 위에서 돔. iOS 앱·안드로이드·IoT 다 HTTP. → [[REST-API]] 참조.

### 챕터 4. 상태코드 I (1xx·2xx·3xx) (슬라이드 29~35)

**키워드 (강의 본문)**:
> 상태 코드 = 클라이언트 요청에 대한 서버 응답 상태
> 1xx (정보): 요청을 받았으며 프로세스를 계속한다
>   100 Continue, 101 Switching Protocol, 102 Processing, 103 Checkpoint
> 2xx (성공): 에러 없이 성공
>   200 OK, 201 Created, 202 Accepted, 203 Non-Authoritative, 204 No Content, 205 Reset, 206 Partial, 207 Multi-Status, 208 Already Reported, 226 IM Used
> 3xx (리다이렉션): 추가 작업 필요
>   300 Multiple Choices, 301 Moved Permanently, 302 Found, 303 See Other, 304 Not Modified, 305 Use Proxy, 307 Temporary Redirect, 308 Permanent Redirect

✦ **강의자 풀이 추정** (LLM 보강)

**상태코드 = 서버가 클라이언트에 보내는 결과 알림 3자리 숫자**. 첫 자리로 5종 분류.

**1xx (정보 응답)** — 거의 안 봄. "받았고 처리 중":
- **100 Continue**: 큰 요청 보내기 전 서버에 확인 받기
- **101 Switching Protocol**: HTTP → WebSocket 같은 프로토콜 전환

**2xx (성공 응답)** ⭐ — **가장 자주 보는 것**:

| 코드 | 이름 | 뜻 | 실사용 |
|---|---|---|---|
| **200** | OK | 요청 성공 | 모든 정상 응답 |
| **201** | Created | 새 자원 생성됨 | POST 후 |
| **202** | Accepted | 받았지만 처리 중 | 비동기 작업 시작 |
| **204** | No Content | 성공했지만 응답 본문 없음 | DELETE 후 |
| **206** | Partial Content | 일부만 응답 | 대용량 파일 부분 다운로드 |

**3xx (리다이렉션)** — "다른 데로 가라":

| 코드 | 이름 | 뜻 |
|---|---|---|
| **301** | Moved Permanently | 영구 이동. 검색엔진 색인 갱신 |
| **302** | Found | 일시 이동. 검색엔진 색인 안 바꿈 |
| **304** | Not Modified | 캐시 그대로 써. 마지막 요청 이후 변경 ❌ |
| **307** | Temporary Redirect | 302와 같지만 메서드 변경 금지 |
| **308** | Permanent Redirect | 301과 같지만 메서드 변경 금지 |

**🎯 이 부분의 핵심 메시지**: 1xx·2xx·3xx는 **"정상" 영역**. 4xx부터 진짜 에러.

⚠️ **흔한 오해**: "301과 302 같은 거". 다름.
- **301**: 영구 이동 → 검색엔진이 새 URL을 색인 ([[SEO]] 중요)
- **302**: 임시 이동 → 검색엔진이 원래 URL 유지
- 사이트 이전 시 **반드시 301** 써야 SEO 안 깨짐.

### 챕터 5. 상태코드 II (4xx·5xx) (슬라이드 36~45)

**키워드 (강의 본문)**:
> 4xx (클라이언트 오류): 요청의 문법이 잘못되었거나 처리할 수 없다
>   400 Bad Request, 401 Unauthorized, 402 Payment Required, 403 Forbidden, 404 Not Found, 405 Method Not Allowed, 406 Not Acceptable, 407 Proxy Auth Required, 408 Request Timeout, 409 Conflict, 410 Gone, 411 Length Required, 412 Precondition Failed, 413 Entity Too Large, 414 URI Too Long, 415 Unsupported Media, 416 Range Not Satisfiable, 417 Expectation Failed, 421 Misdirected, 422 Unprocessable, 423 Locked, 424 Failed Dependency, 426 Upgrade Required, 428 Precondition Required, 429 Too Many Requests, 431 Header Fields Too Large, 451 Unavailable For Legal Reasons
> 5xx (서버 오류): 유효한 요청에 대해 서버가 응답 실패
>   500 Internal Server Error, 501 Not Implemented, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout, 505 HTTP Version Not Supported, 506 Variant, 507 Insufficient Storage, 508 Loop Detected, 510 Not Extended, 511 Network Auth Required

✦ **강의자 풀이 추정** (LLM 보강)

**4xx (클라이언트 오류)** ⭐ — **"네가 잘못함"**:

자주 만나는 것:

| 코드 | 이름 | 뜻 | 실사용 |
|---|---|---|---|
| **400** | Bad Request | 문법 오류 | JSON 깨짐 등 |
| **401** | Unauthorized | 인증 필요 | 로그인 안 됨 |
| **403** | Forbidden | 권한 없음 | 로그인은 했는데 권한 부족 |
| **404** | Not Found | 페이지 없음 | URL 오타 등 (**가장 유명**) |
| **405** | Method Not Allowed | 메서드 불일치 | GET만 지원하는 데 POST 보냄 |
| **408** | Request Timeout | 요청 시간 초과 | |
| **409** | Conflict | 충돌 | 동시 수정 등 |
| **413** | Payload Too Large | 요청 본문 너무 큼 | 파일 업로드 한도 초과 |
| **429** | Too Many Requests | 요청 너무 많음 | Rate Limit |
| **451** | Legal Reasons | 법적 이유로 차단 | 정부 검열 (Orwell 1984 참조한 코드) |

**401 vs 403** — 자주 헷갈림:
- **401 Unauthorized**: "**너 누구야?**" (인증이 안 됨, 로그인 필요)
- **403 Forbidden**: "**누군지는 알겠는데 안 됨**" (인증은 됐지만 권한 부족)

**5xx (서버 오류)** — **"내(서버)가 잘못함"**:

| 코드 | 이름 | 뜻 |
|---|---|---|
| **500** | Internal Server Error | 일반적 서버 에러. 가장 자주 봄 |
| **502** | Bad Gateway | 게이트웨이·프록시 상태 나쁨 |
| **503** | Service Unavailable | 일시적 과부하·서비스 중단 |
| **504** | Gateway Timeout | 게이트웨이 시간 초과 |
| **505** | HTTP Version Not Supported | HTTP 버전 미지원 |

**🎯 이 부분의 핵심 메시지**: 4xx = **클라이언트 잘못**, 5xx = **서버 잘못**. 디버깅 시 첫 자리로 책임 소재 즉시 판단.

⚠️ **흔한 오해**: "500 에러는 항상 서버 잘못". 절반만 맞음. 클라이언트가 이상한 데이터 보내서 서버 코드가 터지면 5xx로 나오지만 **근본 원인은 클라이언트**일 수 있음.

### 챕터 6. CORS / SOP / Same-Origin vs Same-Site (슬라이드 46~55)

**키워드 (강의 본문)**:
> CORS (Cross-Origin Resource Sharing) = 다른 출처 리소스 공유
> SOP (Same-Origin Policy) = 같은 출처에서만 리소스 가져옴
> Origin = Scheme(Protocol) + Host(Domain) + Port
> Simple Request — text/plain 등, GET/POST/HEAD 메서드
> Preflighted Request — OPTIONS로 미리 확인
> Same-Origin: 정확히 일치 (scheme·domain·port 모두)
> Same-Site: scheme/port 다른 서브도메인 상관없음

✦ **강의자 풀이 추정** (LLM 보강) — **08강 가장 어려운 챕터**

**SOP (Same-Origin Policy)** ⭐ = 브라우저의 **기본 보안 정책**.

"같은 출처(Origin)에서 온 리소스만 허용". 그렇지 않으면 차단.

**Origin = Scheme + Host + Port** (셋 다 일치해야):

```
https://www.example.com:443/path/page.html
└─┬───┘  └────┬────────┘ └┬┘
Scheme    Host (Domain)  Port
```

**Same-Origin 판단 예** (기준: `https://www.example.com`):

| URL | 결과 | 이유 |
|---|---|---|
| `https://www.example.com` | ✅ Same-Origin | 정확히 일치 |
| `https://www.example.com:443` | ✅ Same-Origin | 443은 HTTPS 기본 포트 |
| `http://www.example.com` | ❌ Cross-Origin | scheme 다름 (http vs https) |
| `https://example.com` | ❌ Cross-Origin | subdomain 다름 |
| `https://login.example.com` | ❌ Cross-Origin | subdomain 다름 |
| `https://www.example.com:70` | ❌ Cross-Origin | port 다름 |
| `https://www.angel.com` | ❌ Cross-Origin | domain 다름 |

**CORS** ⭐ = SOP의 **예외 허용 메커니즘**.

서버가 명시적으로 "이 출처는 허용함" 헤더를 보내면 브라우저가 통과시킴.

```
서버 응답 헤더:
Access-Control-Allow-Origin: https://www.example.com
```

**CORS Request 2종**:

| | Simple Request | Preflighted Request |
|---|---|---|
| **메서드** | GET·POST·HEAD | PUT·DELETE 등 |
| **Content-Type** | text/plain, form-data, urlencoded만 | 그 외 (application/json 등) |
| **OPTIONS 사전 요청** | 없음 | **있음** ⭐ |
| **속도** | 빠름 | 느림 (왕복 2번) |

**Preflighted 흐름**:
```
1. 브라우저: OPTIONS 요청 (메서드·헤더 사전 확인)
   ↓
2. 서버: "그 출처·메서드·헤더 OK"
   ↓
3. 브라우저: 실제 요청 보냄
```

**Same-Origin vs Same-Site** — 헷갈리는 두 개:

| | Same-Origin | Same-Site |
|---|---|---|
| **기준** | scheme + domain + port 모두 일치 | eTLD+1 (도메인 + .com 한 단계) 일치 |
| **엄격함** | 매우 엄격 | 느슨 |
| **subdomain 차이** | Cross | Same-Site |
| **port 차이** | Cross | Same-Site |
| **scheme 차이** | Cross | Same-Site (Schemeful Same-Site 등장 전) |

→ **Same-Site가 Same-Origin보다 더 넓은 범위**. 쿠키 정책(SameSite 속성)에서 자주 쓰임.

**🎯 이 부분의 핵심 메시지**: SOP는 **브라우저 기본 차단**, CORS는 **명시적 허용**. 웹 보안의 기둥. iOS·Android 앱에는 적용 ❌ (브라우저 정책이라서).

⚠️ **흔한 오해**: "CORS 에러는 서버 잘못". 맞기도 틀리기도. **브라우저가 차단**하는 거지 실제로는 서버가 응답함 — 단 응답을 JavaScript가 못 읽음. 서버에 `Access-Control-Allow-Origin` 추가하면 해결.

⚠️ **흔한 오해 2**: "CORS는 모든 환경에서 작동". **브라우저에서만 작동**. Postman·curl·서버 간 통신은 CORS 무시.

## 💡 일상 비유

### 비유 1: HTTP 상태코드 = 카페 주문 결과

- **2xx 성공** — "주문 받았어요" (200 OK = 정상, 201 Created = 새 메뉴 등록됨, 204 No Content = 됐어요 추가 설명 없음)
- **3xx 리다이렉트** — "옆 매장 가세요" (301 = 영구 이전, 302 = 오늘만)
- **4xx 손님 잘못** — 400 = 주문 이상함, 401 = 회원 카드 가져오세요, 403 = 회원은 맞는데 그 메뉴 권한 없음, 404 = 그런 메뉴 없음, 429 = 너무 많이 시킴
- **5xx 매장 잘못** — 500 = 매장 사정으로 죄송, 503 = 지금 잠시 영업 안 함

### 비유 2: SOP / CORS = 회사 출입 정책

- **SOP**: 회사 보안 정책 = "사내 인트라넷은 사내 직원만"
- **Origin (출처)**: 회사·부서·층 (예: "본사 3층 영업팀")
- **Cross-Origin 요청**: 외부 협력사 직원이 내부 서버 접근 시도
- **CORS 허용**: "이 협력사는 OK"라고 사전에 화이트리스트 등록
- **Preflighted Request**: 출입 전 "내일 OO 출입 가능?" 사전 신청

### 비유 3: TCP/IP 4계층 = 우체국 시스템

- **응용 계층** = 보내는 사람 + 받는 사람 ("선물 보내겠다")
- **전송 계층** = 택배 박스 포장 + 추적번호 + 수령 확인
- **인터넷 계층** = 주소 라벨 + 어느 우체국·허브 거칠지
- **네트워크 액세스** = 실제 트럭·도로
