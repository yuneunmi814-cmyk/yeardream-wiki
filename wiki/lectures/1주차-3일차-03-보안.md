# 03 / 보안 — 교재 버전

> **4섹션 구조** → [[_meta/학습-워크플로우]]
> 08강 [[HTTP-프로토콜]]의 **Stateless 약점을 보완**하는 메커니즘. 인증·인가 + 해시 + 쿠키·세션·토큰·JWT + 캐시·CDN.

## 🚀 예습 (5분용)

### 한 줄 요약

**보안 = 인증(누구) + 인가(무엇 가능) + 상태 저장 4종(쿠키·세션·토큰·JWT) + 무결성 검증(해시) + 성능(캐시·CDN)**. HTTP가 stateless라서 매번 "너 누구?"를 묻지 않으려는 모든 메커니즘.

### 핵심 단어 5~7개

- [[인증과-인가]] ⭐ — Authentication(누구) vs Authorization(무엇 가능)
- [[해시]] — MD5·SHA, 단방향 암호화, 무결성 검증
- [[쿠키]] / [[세션]] — 클라이언트 저장 vs 서버 저장
- [[토큰]] / [[JWT]] ⭐⭐ — 분산 환경의 표준
- [[캐시]] / [[CDN]] — 성능과 보안의 교집합
- **대칭키 vs 비대칭키** — 암호화 양대 방식
- **Refresh Token** — Access Token 만료 보완

### 큰 그림

```
사용자 → 로그인 시도
   ↓
[인증 Authentication] → 비밀번호·생체·OTP·인증서
   ↓ 성공
[인가 Authorization] → 권한 확인 (이 자원 접근 가능?)
   ↓
서버: "OK, 이 세션 키/토큰 줄게"
   ↓
[상태 저장] → 쿠키·세션·토큰·JWT 중 선택
   ↓
다음 요청부터 매번 그 키 첨부 → 서버가 "아 너구나" 인식
```

→ Day 3 마지막 강(4/4). **JWT가 사용자의 솔루션·콘텐츠 자산화 우선순위 1순위** ⭐⭐.

### 챕터 1. 인증 vs 인가 + 암호화 기초 (슬라이드 4~12)

**키워드 (강의 본문)**:
> 인증 (Authentication) — 사용자의 신원을 검증하는 프로세스
> 인증 요소 — 비밀번호·일회용 핀·생체 인식·인증서
> 인가 (Authorization) — 인증된 사용자가 어떤 자원에 접근 가능한지
> 인증과 인가의 필요성 — 민감 정보 보호, 보안 정책 준수
> 암호화 (Encryption) — 정보를 의도하지 않은 수신자가 파악할 수 없는 형태로
> 대칭키 / 비대칭키
> 단방향 (HASH) / 양방향 (비밀키·공개키)
> 블록 암호화 / 스트림 암호화

✦ **강의자 풀이 추정** (LLM 보강)

**[[인증과-인가]] = Authentication vs Authorization** ⭐ — 자주 헷갈리는 2개:

| | 인증 (Authentication) | 인가 (Authorization) |
|---|---|---|
| **질문** | "**너 누구야?**" | "**그거 해도 돼?**" |
| **예** | 로그인 (ID/PW) | 관리자 페이지 접근 권한 |
| **순서** | 먼저 | 인증 후 |
| **결과** | 토큰·세션 ID 발급 | 자원 접근 허용/거부 |
| **줄임** | AuthN | AuthZ |

**인증 요소 4종**:
- **비밀번호** (Knowledge: 아는 것)
- **OTP·인증서** (Possession: 가진 것)
- **생체 인식** — Face ID·지문 (Inherence: 본인인 것)
- **2FA·MFA** — 위 중 2개 이상 조합

**암호화 분류**:

```
암호화
├── 단방향 (해시) — 복호화 불가. 패스워드 저장용
│   └── MD5, SHA-1, SHA-2, SHA-3
│
└── 양방향 — 복호화 가능
    ├── 대칭키 — 암호화·복호화 같은 키. 빠름
    │   ├── 블록 암호화 (AES, DES)
    │   └── 스트림 암호화
    │
    └── 비대칭키 — Public Key(암호화) + Private Key(복호화)
        └── RSA, ECC (SSH·SSL/TLS 핵심)
```

**🎯 이 부분의 핵심 메시지**: **인증 → 인가** 순서. 암호화는 **단방향(해시)**과 **양방향(대칭/비대칭)** 두 갈래.

⚠️ **흔한 오해**: "암호화 = 비밀번호 보호". 그건 **해시**(단방향). 진짜 암호화는 양방향. 패스워드는 복호화 필요 ❌ → 해시가 적합.

---

### 챕터 2. 해시 (슬라이드 13~21)

**키워드 (강의 본문)**:
> 해시 = 임의 길이 데이터를 고정 길이로 매핑
> 해시함수 / 해시값 / 키 / 해싱
> 해시 탄생 배경 — 사이버 범죄·해킹 증가, 민감 정보 암호화 저장
> 해시함수 종류 — 암호화 해시 (MD5·SHA·MAC) / 비암호화 해시 (CRC·Checksum)
> SHA = Secure Hash Algorithm
> SHA-1, SHA-2 (224·256·384·512), SHA-3
> 단방향 암호화 — 역함수 불가, 복호화 불가
> 제1 역상 저항성 — 해시값으로 입력값 찾기 어려움
> 제2 역상 저항성 — 같은 해시값 갖는 다른 입력값 찾기 어려움
> 활용 — 무결성 검증·전자서명·패스워드 검증

✦ **강의자 풀이 추정** (LLM 보강)

**[[해시]] = 임의 길이 데이터 → 고정 길이 출력** ⭐

```
"Hello"      → SHA-256 → 185f8db32271fe25f561a6fc938b2e264306ec304eda518007d1764826381969
"Hello!"     → SHA-256 → 334d016f755cd6dc58c53a86e183882f8ec14f52fb05345887c8a5edd42c87b7
"안녕하세요" → SHA-256 → b15...
```

특징:
- **고정 길이** (입력 길이 무관)
- **단방향** — 결과로 입력 못 찾음
- **민감성** — 1글자 변경 시 완전히 다른 해시
- **무충돌** (이론상)

**해시 알고리즘 종류**:

| 알고리즘 | 출력 길이 | 안전성 | 실사용 |
|---|---|---|---|
| **MD5** | 128 bit | ❌ 깨짐 | 무결성 체크 (보안 ❌) |
| **SHA-1** | 160 bit | ❌ 깨짐 (2017) | 레거시만 |
| **SHA-256** | 256 bit | ✅ 안전 | **현재 표준** |
| **SHA-512** | 512 bit | ✅ 안전 | 고보안 |
| **SHA-3** | 가변 | ✅ 최신 | 미래 표준 |

**3대 활용**:

1. **패스워드 저장** — DB에 평문 ❌, 해시값만. 로그인 시 입력값 해시 → DB 해시와 비교
   ```
   DB: { user: "yoon", password_hash: "5e88..." }
   ```
   유출돼도 원본 패스워드 모름. (단 솔트·반복 해싱 추가 권장)

2. **파일 무결성 검증** — 다운로드 사이트의 SHA256 값으로 변조 확인
   ```
   downloaded.zip의 해시 == 공식 페이지의 해시 → 안전
   ```

3. **전자서명** — Private Key로 메시지 해시를 암호화 → 위변조 확인

**🎯 이 부분의 핵심 메시지**: 해시 = **단방향 + 고정 길이 + 민감**. 패스워드·무결성·서명에 필수.

⚠️ **흔한 오해**: "해시 = 암호화". 다름. 암호화는 **복호화 가능**, 해시는 **불가능**. 그래서 패스워드 저장에 해시 적합 (서버도 원래 비밀번호 몰라야 함).

### 챕터 3. 쿠키 (슬라이드 22~29)

**키워드 (강의 본문)**:
> HTTP의 Stateless — 서버가 클라이언트 상태 안 가짐
> 쿠키 (Cookie) — 서버에서 사용자 브라우저로 전송하는 작은 데이터
> 쿠키 활용 — 추천 영상·알고리즘
> 쿠키 특징 — 도메인당 20개·총 300개·1개 4KB, 클라이언트 저장, 이름·값·만료·경로 정보
> 쿠키 장점 — Stateless에도 이슈 ❌, 로그인 빠름
> 쿠키 단점 — 보안 취약, 용량 제한, 웹 브라우저 형태 차이
> 쿠키 종류 — Session Cookie (브라우저 열린 동안) / Persistent Cookie (유효기간 동안)
> 쿠키 작동원리 — HTTP 요청 → 헤더에 쿠키 응답 → 클라이언트 저장 → 다음 요청에 포함

✦ **강의자 풀이 추정** (LLM 보강)

**HTTP Stateless 약점**:
```
1번 요청: "안녕"               서버: "안녕"
2번 요청: "어 나야"            서버: "나가 누구?"   ← 기억 못 함
```

→ 매번 "내가 누군지" 알려줘야 함. **쿠키가 가장 단순한 해결책**.

**[[쿠키]]** = **클라이언트(브라우저)에 저장되는 작은 데이터** (서버가 발급):

```
서버 → 브라우저:  Set-Cookie: session_id=abc123; Max-Age=3600
브라우저 → 서버:  Cookie: session_id=abc123    (다음 요청부터 자동 첨부)
```

**쿠키 특징**:
- 저장 위치: **클라이언트** (브라우저)
- 크기 제한: **개당 4KB**, 도메인당 20개, **총 300개**
- 구성: 이름·값·만료·경로·도메인
- 자동 첨부: 같은 도메인 요청 시 자동으로 따라감

**쿠키 종류 2종**:
- **Session Cookie** — 브라우저 종료 시 삭제 (메모리)
- **Persistent Cookie** — 만료일까지 유지 (디스크)

**장점**:
- Stateless에도 동작 (서버 부담 ❌)
- 빠른 로그인 (자동 첨부)

**단점**:
- **보안 취약** — 클라이언트에 평문 저장, 탈취 위험
- 용량 제한
- CSRF 공격 표적

**활용 사례**:
- 로그인 유지
- 장바구니
- 추천 알고리즘·광고 추적

**🎯 이 부분의 핵심 메시지**: 쿠키 = **클라이언트 저장 + 자동 첨부 + 4KB**. Stateless 보완의 1세대 솔루션.

⚠️ **흔한 오해**: "쿠키 = 추적·악". 자체로는 중립. **EU GDPR** 이후 동의 요청 의무 (그 "쿠키 동의" 팝업).

### 챕터 4. 세션 (슬라이드 30~35)

**키워드 (강의 본문)**:
> 세션 (Session) — 웹사이트 페이지에 공통 사용되는 사용자 상태 저장
> 서블릿 컨테이너 상태 유지 위한 정보
> 웹 서버 저장 (= 세션 쿠키, 서버 측)
> 각 클라이언트에 고유 Session ID 부여
> 장점 — 쿠키보다 안전, 로그인 유지
> 단점 — 쿠키보다 느림, 서버 자원 사용 → 부하·오버헤드
> 세션 작동원리 — HTTP 요청 → 세션 ID 생성 → 응답(세션 쿠키 포함) → 저장 → 다음 요청 시 첨부 → 서버 인증 → 응답
> 쿠키 vs 세션 — 저장 위치·만료·리소스·용량·속도·보안

✦ **강의자 풀이 추정** (LLM 보강)

**[[세션]]** = **서버에 저장되는 사용자 상태** (클라이언트엔 Session ID만):

```
1. 로그인 → 서버가 Session ID 발급 (서버 메모리·DB에 저장)
   서버: { "session_abc123": { user: "yoon", role: "admin" } }

2. Set-Cookie: session_id=abc123 → 클라이언트 저장 (쿠키와 똑같이)

3. 다음 요청 → Cookie: session_id=abc123 자동 첨부

4. 서버: abc123 조회 → "아, yoon이구나" 인식 → 응답
```

→ **세션도 쿠키를 운반 수단으로 사용** (= "세션 쿠키"). 차이는 **실제 데이터 저장 위치**.

**쿠키 vs 세션** ⭐ — 핵심 비교:

| | 쿠키 | 세션 |
|---|---|---|
| **저장 위치** | 클라이언트 | **웹 서버** |
| **만료** | Set-Cookie의 Max-Age | 브라우저 종료 시 (기본) |
| **리소스** | 클라이언트 | 서버 |
| **용량** | 1개 4KB, 총 300개 | 서버 한도까지 |
| **속도** | 빠름 | 느림 (서버 조회) |
| **보안** | 약함 | **강함** |

**🎯 이 부분의 핵심 메시지**: 세션 = **서버 저장 + Session ID만 쿠키로**. 쿠키보다 안전하지만 서버 부담.

⚠️ **흔한 오해**: "세션은 쿠키 안 씀". 틀림. **Session ID 운반에 쿠키 사용**. 다만 데이터 본체는 서버.

⚠️ **흔한 오해 2**: "세션이 무조건 낫다". 서버 1대면 그렇지만, **여러 서버 분산 환경(스케일 아웃)** 에선 세션 공유 문제 발생 → 토큰·JWT가 등장한 이유.

### 챕터 5. 토큰 (슬라이드 36~40)

**키워드 (강의 본문)**:
> 토큰 (Token) — 서버가 클라이언트를 구별할 수 있도록 발급하는 데이터
> 토큰 유형 — 연결형(USB·스마트카드) / 비접촉형 / 분리형
> 토큰 동작 — 요청 → 인증 디바이스와 통신 → 액세스 확인 → 브라우저에 저장

✦ **강의자 풀이 추정** (LLM 보강)

**[[토큰]]** = **서버가 발급한 인증 데이터**.

세션의 **분산 환경 약점**을 해결하기 위해 등장:

**세션의 약점**:
```
서버 1            서버 2            서버 3
[Session A]      [Session B]      [Session C]
```
사용자 A가 서버 1에 로그인 → 다음 요청이 서버 2로 가면 "Session A 모름" → 재로그인.
해결책: 세션 DB 공유 (느림·복잡) **또는 토큰** (서버에 저장 안 함).

**토큰 흐름**:
```
1. 로그인 → 서버가 토큰 발급 (서명 포함, 서버는 저장 ❌)
2. 클라이언트: 토큰 자기 저장 (LocalStorage·메모리 등)
3. 다음 요청: Authorization: Bearer <토큰> 헤더 첨부
4. 서버: 토큰 서명 검증 → 즉시 인증 (DB 조회 ❌)
```

→ **서버 stateless 유지 + 분산 환경 친화** = 모던 웹·앱의 표준.

**토큰 유형** (강의 본문은 물리 디바이스 위주):
- **연결형**: USB·스마트카드 (물리적 연결)
- **비접촉형**: Microsoft 매직 링 (근거리)
- **분리형**: 2FA OTP (떨어진 디바이스)

→ 실무 웹에서 자주 보는 **소프트웨어 토큰** = 다음 챕터 **JWT**.

**🎯 이 부분의 핵심 메시지**: 토큰 = **서버 저장 ❌, 클라이언트 보유 + 서명 검증**. 분산 환경의 답.

### 챕터 6. JWT (JSON Web Token) ⭐⭐ (슬라이드 41~50)

**키워드 (강의 본문)**:
> JWT = JSON Web Token, 클라이언트-서버 권한 통신 토큰
> JWT 동작 — HTTP 헤더에 실어 서버가 클라이언트 식별
> JSON 데이터 Base64 URL safe Encode → 직렬화, 전자서명 포함
> JWT 구조 — Header.Payload.Signature (aaa.bbb.ccc)
> Header — 토큰 유형 + 해시 알고리즘 (`{"alg":"HS256","typ":"JWT"}`)
> Payload — 사용자 권한 정보·데이터
> Signature — Private Key로 서명, 위변조 검증
> 장점 — 작은 용량·빠른 전송·서버 부담 ↓·권한 데이터·시간 지정 가능
> 단점 — 키 유출 시 시스템 전체 위험·복잡·메시지 전송 못 함
> JWT 동작 — 로그인 → Access Token 발급 → 데이터 요청 시 첨부 → 검증 → 응답
> Refresh Token — Access Token 짧은 만료를 보완, 일주일·2주 만료

✦ **강의자 풀이 추정** (LLM 보강) — **사용자 콘텐츠·솔루션 자산화 1순위** ⭐⭐

**[[JWT]]** = JSON 형식의 인증 토큰. 모던 웹·앱의 사실상 표준.

**JWT 구조 = 점(.)으로 구분된 3 파트**:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9 . eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ . SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
└─────────── Header ────────────┘ └─────────────── Payload ──────────────────────────┘ └────── Signature ──────┘
```

**1. Header** (Base64 인코딩):
```json
{
  "alg": "HS256",     // 서명 알고리즘
  "typ": "JWT"
}
```

**2. Payload** — Claims (주장):
```json
{
  "sub": "1234567890",     // subject (사용자 ID)
  "name": "John Doe",
  "iat": 1516239022,       // issued at
  "exp": 1516242622,       // expiration
  "role": "admin"
}
```

⚠️ **Payload는 Base64만 적용 = 누구나 디코드 가능**. 비밀번호 등 민감 정보 ❌.

**3. Signature**:
```
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  your-256-bit-secret
)
```

서버의 **시크릿 키로 서명**. 클라이언트가 Payload 조작하면 서명 깨짐 → 서버가 거부.

**JWT 동작 흐름**:

```
1. 사용자 로그인 (ID·PW)
   ↓
2. 서버 인증 후 JWT 발급
   ↓
3. 클라이언트: 토큰 저장 (LocalStorage·sessionStorage·HttpOnly Cookie)
   ↓
4. 매 요청: Authorization: Bearer <JWT>
   ↓
5. 서버: 서명 검증만 (DB 조회 ❌) → 즉시 인증
   ↓
6. 데이터 응답
```

**JWT 장점**:
- **작고 빠름** (Base64 텍스트)
- 서버 저장 ❌ → **스케일 아웃 친화**
- 권한·만료 시간 등 정보 내장
- 다양한 언어·플랫폼 호환 (JSON 표준)

**JWT 단점**:
- **시크릿 키 유출 시 시스템 전체 위험** ⚠️
- 한 번 발급된 토큰은 **취소 어려움** (서명만 검증)
- Payload 평문 (민감 정보 ❌)
- 토큰 자체 길이가 쿠키보다 큼

**Refresh Token** ⭐ — JWT 단점 보완:

```
[Access Token]  만료 짧음 (예: 15분)  ← 자주 노출되니까 위험 줄임
[Refresh Token] 만료 김 (예: 2주)     ← 안전한 곳에 보관

흐름:
1. 로그인 → Access (15분) + Refresh (2주) 둘 다 발급
2. Access 만료 → Refresh로 새 Access 발급 (재로그인 ❌)
3. Refresh 만료 → 그때만 재로그인
```

**🎯 이 부분의 핵심 메시지**: JWT = **자기 충족 인증 토큰** (서명만 검증, DB 조회 ❌). 분산 환경의 표준. Refresh Token으로 만료 보완.

⚠️ **흔한 오해**: "JWT는 암호화". **서명만**. Payload는 누구나 디코드 가능. 민감 정보 넣지 말 것.

⚠️ **흔한 오해 2**: "JWT가 무조건 세션보다 낫다". 작은 서비스·단일 서버는 **세션이 더 안전·간단**. JWT는 분산·다플랫폼 시 진가.

**실무 적용** (사용자 프로젝트윤365 관점):
- iOS 앱 ↔ 백엔드 인증 → JWT
- 클라이언트 솔루션의 API 인증 → JWT
- 단일 페이지 웹 (SPA) → JWT + Refresh

### 챕터 7. 캐시 / CDN (슬라이드 51~60)

**키워드 (강의 본문)**:
> 캐시 (Cache) — 자주 필요한 데이터·값 복사본을 임시 저장
> 캐시 특징 — IOPS 증가·DB 비용 절감·앱 성능 개선·백엔드 부하 감소
> 캐싱 (Caching) — 오래 걸리는 작업 결과를 저장
> 캐시 작동 — RAM 같은 빠른 하드웨어에 저장, 디스크 액세스 줄이기
> 캐시 vs 쿠키 — 임시 저장(웹 요소) vs 인증 정보
> CDN (Content Delivery Network) — 고용량 데이터를 어느 위치에서든 안정적 전송
> CDN 구성 — Origin Server + CDN Cache Server 분산 배치
> CDN 장점 — 글로벌 로딩 속도·비용 절감
> CDN 단점 — 연결 지점 증가·SPOF(단일 장애점) 위험
> 캐시 종류 — 콘텐츠 캐시·클라우드 스토리지·멀티미디어 스트리밍

✦ **강의자 풀이 추정** (LLM 보강)

**[[캐시]] = 자주 쓰는 데이터를 가까이·빠른 곳에 임시 저장**.

```
요청 → 캐시 확인
       ├── 있음 (Hit)  → 즉시 응답 (빠름)
       └── 없음 (Miss) → 원본에서 가져옴 → 캐시 저장 → 응답
```

**캐시 위치별**:
- **CPU 캐시** (L1·L2·L3): CPU 내부 (Day 2 [[1주차-2일차-02-메모리]] 참조)
- **메모리 캐시**: Redis·Memcached (DB 부담 완화)
- **브라우저 캐시**: 이미지·CSS·JS (재방문 시 빠름)
- **CDN 캐시**: 지리적 분산 (다음 챕터)

**캐시 vs 쿠키** — 비슷한데 다름:

| | 캐시 | 쿠키 |
|---|---|---|
| **목적** | 성능 (빠른 로딩) | 식별·인증 |
| **데이터** | 페이지·이미지·CSS | 사용자 정보 |
| **삭제** | 수동 (또는 만료) | 만료 기간 자동 |
| **예시** | 자주 본 이미지 | 로그인 정보 |

**[[CDN]] (Content Delivery Network)** ⭐ — 캐시의 **지리적 분산** 버전:

```
Origin Server (서울)
       ↑
       │
   ┌───┼───────────────────┐
   │   │                   │
[Tokyo Cache] [Singapore] [LA Cache]
   ↓           ↓           ↓
일본 사용자  싱가폴 사용자  미국 사용자
```

→ 사용자가 **가장 가까운 캐시 서버**에서 받음. 응답 빠름 + 원본 서버 부담 ↓.

**CDN 실사용**:
- Cloudflare (대표적, 사용자 일상 사용)
- AWS CloudFront
- Netflix Open Connect (자체 CDN)
- 한국 통신사 콘텐츠 캐시 서버 (KT·SKB)

**CDN 장점**:
- 글로벌 로딩 속도 ↑
- 원본 서버 비용 ↓
- DDoS 방어 (Cloudflare 같은 곳)

**CDN 단점**:
- **단일 장애점 (SPOF)**: Cloudflare 다운 → 전 세계 사이트 마비 (2022년 실제 발생)
- 비용 (트래픽 많을수록)
- 캐시 무효화 복잡

**🎯 이 부분의 핵심 메시지**: 캐시 = **자주 쓰는 거 가까이 두기**. CDN = **그걸 지구 단위로 분산**.

⚠️ **흔한 오해**: "캐시 = 영원 저장". **만료** 있음 (TTL). 잘못 설정하면 오래된 데이터 응답.

## 💡 일상 비유

### 비유 1: 인증 vs 인가 = 회사 출입 절차

- **인증 (Authentication)**: 정문에서 **사원증 확인** ("너 직원 맞아?")
- **인가 (Authorization)**: 9층 임원실 카드키 ("**임원만** 들어와")

직원이라는 건 인증, 임원이라는 건 인가. **순서가 중요** — 정문 통과 못 하면 9층 권한 따질 필요도 없음.

### 비유 2: 쿠키 vs 세션 = 휴대폰 vs 영수증

- **쿠키 (영수증)**: 손님이 **직접 들고 다님**. 잃어버리면 끝. 위조 가능. → 가볍지만 안전 ❌
- **세션 (휴대폰 회원번호)**: 가게가 **자기 DB**에 기록. 손님은 회원번호만 들고 다님. 회원번호 잃어도 가게가 다시 발급. → 안전하지만 가게 부담

### 비유 3: JWT = 출입 스티커 (위변조 방지 시일)

영화관 손목 밴드:
- **Header**: 어느 영화관 (HS256 = 어느 보안 회사)
- **Payload**: 좌석 번호·시간 (사용자 정보) — **누구나 볼 수 있음**
- **Signature**: 영화관 도장 (시크릿 키 서명) — **위조하면 들통**

손님이 좌석을 마음대로 바꿔도 도장이 안 맞아서 입장 거부. **서버는 도장만 확인**, 손님 명단 DB 조회 ❌ → 빠름.

### 비유 4: CDN = 전국 편의점

서울 본사(Origin)가 전국 손님 다 응대하면 죽음. **편의점(CDN Cache)을 동네마다** 두면:
- 손님은 동네 편의점에서 빠르게 사옴
- 본사 부담 ↓
- 단 편의점 망하면 (Cloudflare 다운) 동네 전체 마비
