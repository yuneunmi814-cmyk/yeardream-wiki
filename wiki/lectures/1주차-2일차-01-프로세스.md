## 🚀 예습 (5분용)

### 한 줄 요약

**프로세스 = 실행 중인 프로그램**. 메모리 4영역(코드·데이터·힙·스택), 5상태 전이(생성→준비→실행→대기→종료), 멀티프로세싱 + IPC, 스케줄링, 교착/기아.

### 핵심 단어 5~7개

- [[운영체제]] — OS 통합
- **프로세스 vs 스레드** — 독립 메모리 vs 공유 메모리
- **메모리 4영역** — 코드·데이터·힙·스택
- **5상태 전이** — 식당 손님 비유
- **IPC** — Inter Process Communication
- **스케줄링** — FCFS·SJF·라운드 로빈
- **교착상태(Deadlock) vs 기아(Starvation)**

### 큰 그림

```
프로세스 = 실행 중 프로그램
   ↓
┌─ 구조: 코드·데이터·힙·스택 4영역
├─ 상태: 생성→준비⇄실행→대기→종료 (식당 비유)
├─ 다중처리: 멀티스레드(공유) / 멀티프로세스(격리) + IPC
├─ 스케줄링: FCFS·SJF·RR
└─ 문제: 교착(서로 기다림) / 기아(낮은 우선순위)
```

→ Day 2 8강 중 **2강**. OS 4대 기능 첫 번째 (프로세스 관리).

## 📖 본문 (정독용)

### 챕터 1. 프로세스의 개념 (슬라이드 1~7)

**키워드 (강의 본문)**:
- **프로세스** = 실행 중인 프로그램
- **프로그램** = 특정 작업 수행 명령어 모음. 실행되면 프로세스
- **스레드** = 프로세스 안에서 실행되는 흐름 단위. 1개 이상 존재
- 프로세스 vs 스레드: **프로세스는 독립적 메모리**, 스레드는 **프로세스 메모리 공유**

✦ **강의자 풀이 추정**
- **프로그램** = 디스크의 정지 파일 (`Slack.app`)
- **프로세스** = 메모리에서 실행 중 (Slack 켜진 상태)

| | 메모리 | 통신 | 무게 |
|--|--------|------|------|
| 프로세스 | 독립 | 어려움(IPC) | 무거움 |
| 스레드 | 공유 | 쉬움(변수 직접) | 가벼움 |

→ Chrome 탭 = 보통 별개 프로세스 (안정성·보안). 카카오톡 = 한 프로세스 + 여러 스레드 (메시지·다운로드·알림).

**🎯 핵심 메시지**: 프로세스 = 실행 중 프로그램. 스레드 = 프로세스 안 흐름. 메모리 격리 vs 공유.

⚠️ **흔한 오해**: "프로그램 = 프로세스" → 거짓. 프로그램은 정지, 프로세스는 실행.

---

### 챕터 2. 프로세스 메모리 구조 (슬라이드 8~20)

**키워드 (강의 본문)**:
```
스택 (Stack)   ← 함수 호출·로컬 변수
힙   (Heap)    ← 동적 할당 데이터
데이터 (Data)  ← 전역·정적 변수
코드 (Code)    ← 실행 명령. 읽기 전용
```
- **컴파일**: 기계어 변환. 컴파일러가 한다
- **정적 할당** = 실행 전 미리. 빠르지만 낭비. 학생 45자리 미리 잡기
- **동적 할당** = 실행 중 필요할 때. 정확하지만 지연
- **메모리 누수** = 동적 메모리 해제 안 함 → 성능 저하
- **스택 자료구조** = LIFO(후입선출). push/pop
- **매개변수**: 함수 입력값. 함수 호출 시 스택에 저장. 종료 시 소멸

✦ **강의자 풀이 추정**
4영역 의미:
- **코드**: 영화관에서 상영 중 영화 파일 (안 바뀜)
- **데이터**: 책상 서랍의 자주 쓰는 도구 (프로그램 내내)
- **힙**: 그때그때 꺼내는 메모지 (직접 관리, 안 풀면 누수)
- **스택**: 임시 자료 쌓기 (함수 끝 자동 소멸)

**정적 vs 동적**:
- 식당 좌석 50석 미리 (정적) vs 손님마다 테이블 펴기 (동적)

**🎯 핵심 메시지**: 메모리 4영역 = 코드·데이터·힙·스택. 힙(누수 위험) vs 스택(자동 정리).

⚠️ **흔한 오해**: "Python·Swift엔 메모리 누수 없음" → 거짓. GC·ARC도 완벽하지 않음. 순환 참조 등 누수 가능.

---

### 챕터 3. 프로세스 5상태 ⭐ (슬라이드 21~28)

**키워드 (강의 본문 — 식당 비유)**:
- **생성(new)**: 번호표 들고 식당 밖에서 대기
- **준비(Ready)**: 식당에 들어가 앉음. CPU 차례 대기
- **실행(Running)**: 나온 요리를 먹는 중
- **대기(Blocked/Waiting)**: 화장실 다녀오는 중. I/O 등 기다림
- **종료(Terminated)**: 일어나 나갈 준비

```
[생성] → [준비] ⇄ [실행] → [종료]
              ↑      ↓
              └─ [대기] ←
```

✦ **강의자 풀이 추정**
사용자 맥북 활성 상태 보기에서:
- 50~100개 프로세스 = 대부분 **준비** 상태
- CPU 코어 8개 = 한 번에 **실행** 8개만
- 나머지는 빠른 전환 (타임 슬라이스)

**컨텍스트 스위치**: 한 프로세스 → 다른 프로세스 전환. 수십 밀리초 단위. 사용자 눈엔 동시.

→ Day 1 [[lectures/1주차-1일차-01-앱-웹-서비스|2강]] 동기/비동기와 연결. 비동기 = 프로세스가 대기 상태 안 가고 다음 일.

**🎯 핵심 메시지**: 5상태 = 생성·준비·실행·대기·종료. 식당 손님 비유로 한 번에.

⚠️ **흔한 오해**: "한 번에 하나만 CPU 사용" → 부분적. **CPU 코어가 8개면 동시 8개**. 각 코어는 하나씩.

---

### 챕터 4. 멀티프로세싱 + IPC (슬라이드 29~40)

**키워드 (강의 본문)**:
- **멀티 프로세싱** = 여러 프로세스 동시 처리
- **멀티 스레드** = 한 프로세스 안 여러 작업. 빠르지만 **한 스레드 문제 = 전체 영향**
- **멀티 프로세스** = 여러 프로세스. 독립. Chrome 탭별
- **IPC** (Inter Process Communication) 3종:
  - **메시지 큐**: 공용 우편함에 넣어두기
  - **공유 메모리**: 현관문 열어두기. 빠르지만 충돌 위험
  - **소켓**: 택배로 보내기. **네트워크 너머 가능**

✦ **강의자 풀이 추정**
**프로세스 vs 스레드 비유**:
- 멀티 스레드 = 한 집안 가족이 각자 일 (요리·청소·빨래)
- 멀티 프로세스 = 여러 집이 각자 일

**IPC 일상**:
- 메시지 큐 = 인스타 DM (보내고 받는 사람이 안 켜져 있어도 OK)
- 공유 메모리 = 노션 공유 페이지 (동시 편집)
- 소켓 = HTTP (Claude API 호출도 소켓 통신)

**🎯 핵심 메시지**: 멀티 스레드(공유·빠름) vs 멀티 프로세스(격리·안전). IPC로 안전 통신.

⚠️ **흔한 오해**: "Chrome이 RAM 많이 먹어서 안 좋음" → 부분적. 탭별 프로세스로 안정성·보안 강화. 트레이드오프.

---

### 챕터 5. 스케줄링 + 교착·기아 (슬라이드 41~52)

**키워드 (강의 본문)**:
- **스케줄링**: 어떤 손님부터 처리할지. 대기 최소화 + 공평
- **FCFS** (선입선처리): 먼저 온 순. 단순, 응답성 부족
- **SJF** (최단작업우선): 평균 대기 최소. 단 긴 작업 **기아**
- **라운드 로빈(RR)**: 일정 시간씩 순서. 응답성 좋음. 전환 비용
- **교착상태(Deadlock)**: 서로 기다림 → 아무것도 완료 X. 신입사원 무한루프
- **4조건**: 상호배제·점유대기·비선점·순환대기. 하나만 막아도 예방
- 해결: **은행원 알고리즘**·**무시**·**프로세스 종료**
- **기아상태(Starvation)**: 낮은 우선순위라 자원 못 받음. **Aging**으로 해결

✦ **강의자 풀이 추정**
**현대 OS는 RR 변형 기반**. macOS·Linux·Windows 모두.

**교착상태 일상 비유**: 일방통행 좁은 길에서 두 차 마주침. "네가 먼저 비켜" → 영원히 정지. 철학자의 식사 문제도 같은 구조.

**기아 해결 (Aging)**: 오래 기다린 프로세스 우선순위 자동 상승. 결국 모두 처리됨.

→ 사용자 맥북 가끔 멈춤 → 강제 종료 = 교착상태일 가능성.

**🎯 핵심 메시지**: 스케줄링 3종(FCFS·SJF·RR). RR이 현대 표준. 교착 vs 기아 구분.

⚠️ **흔한 오해**: "OS가 교착상태 다 알아서 막음" → 거짓. 일부 OS는 **Ostrich Algorithm**(무시) 적용. 그래서 가끔 앱 멈춤.

## 💡 일상 비유

### 비유 1 — 카카오톡 멀티 스레드

한 카카오톡 앱 안에서 동시에: 메시지 받기 + 사진 다운로드 + 입력창 표시 + 알림 사운드. 한 프로세스(카톡) 안 여러 스레드.

### 비유 2 — 일방통행에서 마주친 두 차 (교착상태)

두 차가 좁은 길에서 마주침. 둘 다 "네가 먼저 비켜" → 영원히 멈춤. 한 명 양보하지 않으면 둘 다 못 감. 컴퓨터의 영원한 문제.
