### 핵심 단어 5~7개

- [[라이브러리-vs-프레임워크]] — 통합 노트

- **프레임워크** = 빵틀
- **팀 버너스리** — www 창시자
- **Chart.js·Passport.js·Django** — 대표 예시
- **라이브러리 vs API** — 내 컴퓨터 vs 네트워크 너머
- **IoC** (보강) — 호출 방향 역전

        ↓
┌──────────────────────┐

## 📖 본문 (정독용)

### 챕터 1. 웹의 역사 (슬라이드 1~6)

**키워드 (강의 본문)**:
- 웹의 기본 구조: 브라우저(Chrome·Firefox·Brave) ⇄ HTML ⇄ 웹서버
- **세계 최초 웹사이트**: http://info.cern.ch (1991, 지금도 살아있음)
- **팀 버너스리** = www 창시자. 영국 물리학자. **HTML·CSS·HTTP·URL** 모두 그의 작품
- 연구자 자료 공유용 → **대중에게 무료로 공개**. 기사작위·서울 평화상
- **1998 vs 2008 네이버**: 단순 검색 → 복합 포털 (10년 만에 정보 밀도 폭증)
- → 결과: 반복 코드·협업 증가·구조화 필요 = **라이브러리·프레임워크 등장 배경**

✦ **강의자 풀이 추정**
**팀 버너스리 1993년 CERN의 무료 공개 결정** = 현대 인터넷의 출발점. 만약 특허 걸었다면 인스타·유튜브·구글 다 다른 모습(또는 부재).

→ 사용자가 [[프로그램-개요|이어드림스쿨]]에서 위키 만드는 것도 이 결정의 후손.

→ 웹 복잡화 = 사용자 인스타 2010 vs 2026 비교와 같은 패턴.

**🎯 핵심 메시지**: 웹 복잡화가 라이브러리·프레임워크 등장의 동기. 30년 전 무료 공개 결정이 토대.

⚠️ **흔한 오해**: "GNU와 Linux는 같은 것" → 거짓. GNU는 도구, Linux는 커널. 합쳐서 GNU/Linux.

### 챕터 2. 라이브러리 (슬라이드 7~13)

**키워드 (강의 본문)**:
- **정의**: 반복해서 자주 사용하는 코드 모음 / 도구 모음
- 다른 이름: **package**
- 오픈소스 영역에서 활발히 생산·공유
- **예시 1 — Chart.js**: 색·데이터 수치만 입력해서 복잡한 Chart 쉽게
- **예시 2 — Passport.js**: 로그인·회원가입·소셜 로그인(카카오·네이버) 라이브러리
- 검증된 라이브러리 활용도 개발자 능력. **잘못 사용 시 성능 저하**
- 한 문장 요약: **생산성 향상·효율성 증진**

✦ **강의자 풀이 추정**
언어별 패키지 저장소:
- JavaScript: **npm** (200만+ 라이브러리)
- Python: **PyPI** (50만+)
- Swift: CocoaPods·SPM
- Flutter: pub.dev

**Passport.js로 카카오 로그인 30분 만에 붙임**. 직접 구현하면 OAuth·해싱·세션 다 짜야 함 (수천 줄).

**🎯 핵심 메시지**: 라이브러리 = 남이 만든 도구 가져다 씀. 선택 자체가 스킬.

⚠️ **흔한 오해 1**: "라이브러리 = 무조건 무료" → 라이선스 확인 필수 (MIT·Apache·GPL·상용).

⚠️ **흔한 오해 2**: "라이브러리 많이 쓸수록 좋은 개발자" → 꼭 필요한 것만. 의존성 지옥·악성 코드(2021 event-stream 사건 등) 위험.

📎 자세히: [[라이브러리-vs-프레임워크]], [[현업-용어]] 오픈소스

### 챕터 3. 라이브러리 vs API ⭐ (슬라이드 14)

**키워드 (강의 본문)**:
| | 라이브러리 | API |
|--|---|---|
| 공통 | 재사용 가능 코드의 반복 활용 | 동일 |
| 차이 | (내 컴퓨터) | **네트워크·통신을 통해 기능 제공** |

✦ **강의자 풀이 추정**
```python
# 라이브러리 (내 컴퓨터)
import numpy
result = numpy.mean([1, 2, 3])  # 0.001초

# API (네트워크 너머)
import requests
result = requests.get("https://api.example.com/mean?...").json()  # 0.1초
```

같은 평균 계산. 빠름·무료(라이브러리) vs 외부 데이터·비용 발생(API).

사용자 일상 예:
- **라이브러리 활용**: Claude Code 안에서 numpy 사용
- **API 활용**: Claude Code → Claude API 호출 (= 우리와의 대화)

**🎯 핵심 메시지**: 라이브러리(내 컴퓨터) vs API(네트워크 너머). 같은 재사용, 다른 위치.

📎 자세히: [[REST-API]] (API), [[라이브러리-vs-프레임워크]] (라이브러리)

### 챕터 4. 프레임워크 + 빵틀 비유 ⭐ (슬라이드 15~16)

**키워드 (강의 본문)**:
- **정의**: 프로그램을 효율적으로 짜기 위한 **틀(Frame)**
- **빵틀 비유**: 빵 모양을 만들기 위한 빵틀처럼 **코드 작성 구조 제시**
- 공통 규칙 활용 → **협업 매우 유리**
- 언어별로 다양한 프레임워크. 제공 도구·툴 다름

✦ **강의자 풀이 추정**
**빵틀 = 모양 강제 + 반죽(로직)만 채움**. 라이브러리·프레임워크의 결정적 차이:

| | 라이브러리 | 프레임워크 |
|--|---|---|
| 호출 방향 | 내가 부른다 | **프레임워크가 나를 부른다** |
| 비유 | 도구 모음 | 빵틀 |
| 자유도 | 높음 | 낮음 (구조 강제) |
| 협업 효율 | 낮음 | 높음 (모두 같은 구조) |

→ 사용자 SwiftUI = 프레임워크 (`App.swift`·`ContentView.swift` 자동 생성, 그 안에 채워 넣음).

**🎯 핵심 메시지**: 프레임워크 = 빵틀. 구조 강제 = 협업의 이점.

⚠️ **강의 본문 보완 — IoC**: 강의 본문엔 **IoC(Inversion of Control)·Hollywood Principle** 단어가 없음. 빵틀 비유와 같은 개념의 CS 표준 용어:

> "Don't call us, we'll call you" — 프레임워크의 본질

AI Agent 프레임워크(LangChain) vs 라이브러리(OpenAI SDK) 선택 시 IoC 관점이 유용.

### 챕터 5. 프레임워크 예시: Django (슬라이드 17~21)

**키워드 (강의 본문)**:
- **Django**: Python 기반 웹 프레임워크
- **상당한 수준의 관리자 페이지 자동 제공**

✦ **강의자 풀이 추정**
Django 사용 회사: 인스타그램·핀터레스트·Spotify 일부.

Django의 빵틀:
- 폴더 구조 자동 (`models/`·`views/`·`templates/`)
- DB 모델 정의 → **Admin 페이지 자동 생성** (운영자가 코드 없이 CRUD)
- CSRF·XSS 기본 탑재
- 인증·세션·캐싱 내장

→ 직접 만들면 1주일+. Django는 모델 정의만으로 끝.

→ 사용자 Stage 3 AI 서비스개발자 트랙 가면 Django·FastAPI 마주칠 가능성.

**🎯 핵심 메시지**: Django = Python 표준 웹 프레임워크. Admin 자동 생성이 결정적.

⚠️ **흔한 오해**: "React는 라이브러리야 프레임워크야?" → 공식 라이브러리. 단 Next.js·Remix 같은 메타프레임워크 포함 시 프레임워크급.
